# 資料庫更新通知機制的方案總覽

你要在多台桌面程式之間同步 EF6 資料庫更新並刷新快取，本質上是「跨進程通知機制」。以下是主要可行架構：

## 方案一：資料庫層級觸發 + 通知機制

1. 使用資料庫觸發器偵測 INSERT / UPDATE / DELETE
2. 觸發器寫入一個「變更記錄表」或「版本號」
3. 程式週期性檢查版本號是否變動
4. 若變動則重新讀取快取

優點：簡單、無額外服務
缺點：輪詢仍有延遲、資料庫壓力略增

## 方案二：SQL Server Query Notification（僅 SQL Server 支援，不支援 SQLite / MySQL）

1. 使用 SqlDependency 監聽資料變動
2. 系統自動觸發 OnChange 事件
3. 程式收到事件後刷新快取

優點：事件驅動、低延遲
缺點：僅支援 SQL Server 且資料表需要特定條件（必須可用於 Service Broker）

## 方案三：建立中介服務（推薦通用做法）

1. 建立一個後端服務 (Web API / Windows Service)
2. 所有程式透過這個服務存取資料庫
3. 服務內部維護快取並廣播變更（例如 WebSocket / SignalR Hub / gRPC）
4. 桌面程式連接此服務，收到推播後更新本地快取

優點：

- 真正即時（類似 SignalR）
- 所有快取邏輯集中、易維護
- 可加入權限、記錄、交易控制

缺點：

- 需要額外部署服務

## 方案四：使用訊息佇列（Message Queue）

1. 程式在更新資料時，除了 SaveChanges() 也送出一則訊息至 MQ（如 RabbitMQ、Redis Pub/Sub、Azure Service Bus）
2. 其他程式訂閱此訊息
3. 收到訊息後刷新快取

優點：高可用、高擴充性、事件驅動
缺點：需要額外安裝 MQ

## 方案五：P2P 廣播（同網段）

1. 每個程式啟動一個 UDP Listener
2. 當某程式更新資料時，發送 UDP 廣播封包
3. 其他程式收到後刷新快取

優點：不需額外伺服器
缺點：僅限同區網、不可靠封包、需防止自觸發循環

## 結論建議（依系統規模）

1. 小規模（幾台電腦、同網段） → UDP 廣播 / 輪詢版本號
2. 中規模（10+ 用戶、多地點） → 中介服務 + WebSocket / SignalR
3. 大規模、雲端 → MQ 或微服務架構

## 延伸設計要點

1. Cache 要可區分「資料類型」，不必全清
2. 更新時使用「版本號」或「時間戳」比純 boolean 更好
3. EF6 無內建 Change Tracking for external processes
4. 若你已使用 Autofac，可將 “CacheManager” 設為 Singleton，並由通知機制觸發其刷新

# 目前環境建議

## 系統環境條件整理

1. 多台桌面程式
2. 分佈在不同辦公室／廠房
3. 廠房之間應該有內部網路連線
4. 目前資料庫是 SQLite
5. 未來可能改 SQL Server（不確定）
6. 不會 ASP.NET（暫時不想自己寫 Web 服務）

## SQLite 的限制

1. 不支援 SqlDependency 或 Query Notification
2. 沒有原生事件推播機制
3. 多用戶同時寫入效能有限（寫入時會鎖整個 DB）
4. SQLite 適合「單機或小規模共享」，不適合「多據點同時寫入大量資料」

## 因為你不確定是否會換 SQL Server → 架構要「可替換資料庫」

所以必須避免依賴 SQL Server 專屬功能（例如 SqlDependency）

## 最務實的演進式解法

### 第一階段（立即可做、最簡單）

定期輪詢資料版本（例如每 30 秒檢查一次資料庫中的「版本表」）

1. 建一張 VersionTable（Key, LastUpdateTime 或 VersionNumber）
2. 每次資料寫入時，同時更新該表
3. 各桌面程式每隔 X 秒檢查版本是否有變
4. 有變 → 更新快取

優點：不受資料庫種類限制（SQLite 可用，之後 SQL Server 也可用）
缺點：有延遲、略微增加 DB 壓力（可接受）

→ 若你現在想先上線，又不想改太多架構，這是最實用起點

### 第二階段（需要較即時、可跨據點）

加一個「輕量中介服務」，不一定要 ASP.NET，可以：

選項 A：使用 Console App + TCP/UDP + 自訂協定
選項 B：使用 gRPC（比 Web API 更輕量、更適合桌面程式）
選項 C：使用 SignalR Self-Host（不一定需要 IIS 或 ASP.NET MVC，可用 OWIN 自架 Host）

中介服務負責：

1. 控制資料庫存取（未來換 DB 不影響桌面程式）
2. 維護 Cache
3. 廣播更新事件給所有 Client

桌面程式連接到中介服務：

- WebSocket / TCP / gRPC
- 收到「更新通知」後刷新快取

優點：

- 架構清晰、集中控制
- 可加入權限/記錄/交易
- 可支援 SQLite 或 SQL Server（中間層實作抽象化）
- 可逐步導入，不需一開始就完全改掉 EF6

### 第三階段（高可靠、高擴充）

導入 Message Queue（RabbitMQ / Redis PubSub / Azure Service Bus）

- 發布「資料更新事件」
- 所有程式訂閱此事件

→ 適用大型系統，可能你暫時不需要

# 第一階段：版本表 + 輪詢 的可執行範例架構

以下提供「第一階段：版本表 + 輪詢」的可執行範例架構，包含：

1. 資料表設計（VersionTable）
2. EF6 的 DbContext 與 SaveChanges 擴充（寫入資料時同步更新版本）
3. CacheManager（區域快取 + 事件通知）
4. PollingService（背景輪詢 + 非同步取消 + 去抖）
5. Autofac 註冊範例
6. WinForms 客戶端如何訂閱並在 UI 執行緒更新快取
    程式碼皆為最小可用範例，方便直接貼入專案編譯執行。

## 架構流程（簡要）

1. 資料更新（Repository 呼叫 SaveChanges）
2. SaveChanges 在 DB 中更新資料並同時更新 VersionTable 的對應鍵（Key）
3. 各桌面程式執行 PollingService，定期檢查 VersionTable 是否變更
4. 若變更，PollingService 觸發 CacheManager 下載最新資料並發出 CacheUpdated 事件
5. UI 訂閱事件並在 UI 執行緒更新畫面（使用 SynchronizationContext）

## 資料表設計

1. VersionTable：Key（string，代表資料類別）、VersionNumber（long）或 LastUpdated（DateTime）
2. 使用 Key 可做到部份快取失效（只刷新受影響類別）

## 程式碼：資料模型（VersionEntry）

```csharp
public class VersionEntry
{
    public int Id { get; set; }
    public string Key { get; set; } // e.g. "Products", "Users"
    public long VersionNumber { get; set; }
    public DateTime LastUpdated { get; set; }
}
```

## 程式碼：EF6 DbContext（AppDbContext）

說明：覆寫 SaveChanges，當執行到需通知的更新時，呼叫 UpdateVersion(key)。

```csharp
using System;
using System.Data.Entity;
using System.Linq;

public class AppDbContext : DbContext
{
    public AppDbContext() : base("name=AppDbContext") { }
    public DbSet<VersionEntry> VersionEntries { get; set; }
    public DbSet<Product> Products { get; set; } // 範例資料表

    // 呼叫於 Repository 在變更後
    public void TouchVersion(string key)
    {
        var entry = VersionEntries.SingleOrDefault(v => v.Key == key);
        if (entry == null)
        {
            entry = new VersionEntry { Key = key, VersionNumber = 1, LastUpdated = DateTime.UtcNow };
            VersionEntries.Add(entry);
        }
        else
        {
            entry.VersionNumber += 1;
            entry.LastUpdated = DateTime.UtcNow;
            Entry(entry).State = EntityState.Modified;
        }
        SaveChangesInternal();
    }

    // 內部 SaveChanges（避免遞迴）
    private int SaveChangesInternal()
    {
        return base.SaveChanges();
    }

    // 你通常不直接覆寫 SaveChanges() 避免影響 EF 行為，改用 Repository 在寫入後呼叫 TouchVersion。
}
```

## 程式碼：Repository 範例（在修改資料時同時更新版本）

```csharp
public class ProductRepository
{
    private readonly Func<AppDbContext> _dbFactory;

    public ProductRepository(Func<AppDbContext> dbFactory)
    {
        _dbFactory = dbFactory;
    }

    public void AddProduct(Product p)
    {
        using (var db = _dbFactory())
        {
            db.Products.Add(p);
            db.SaveChanges();
            db.TouchVersion("Products");
        }
    }

    public void UpdateProduct(Product p)
    {
        using (var db = _dbFactory())
        {
            db.Entry(p).State = System.Data.Entity.EntityState.Modified;
            db.SaveChanges();
            db.TouchVersion("Products");
        }
    }
}
```

## 程式碼：CacheManager（Singleton，有事件）

說明：CacheManager 暫存資料，並提供 Refresh(key) 方法與事件 CacheUpdated。

```csharp
using System;
using System.Collections.Concurrent;
using System.Threading.Tasks;

public class CacheUpdatedEventArgs : EventArgs
{
    public string Key { get; }
    public CacheUpdatedEventArgs(string key) { Key = key; }
}

public class CacheManager
{
    private readonly ConcurrentDictionary<string, object> _cache = new ConcurrentDictionary<string, object>();
    public event EventHandler<CacheUpdatedEventArgs> CacheUpdated;
    private readonly Func<AppDbContext> _dbFactory;

    public CacheManager(Func<AppDbContext> dbFactory)
    {
        _dbFactory = dbFactory;
    }

    public T Get<T>(string key) where T : class
    {
        if (_cache.TryGetValue(key, out var v)) return v as T;
        return null;
    }

    public async Task RefreshAsync(string key)
    {
        // 根據 key 決定要載入哪些資料；範例以 Products 為例
        if (key == "Products")
        {
            using (var db = _dbFactory())
            {
                var data = await Task.Run(() => db.Products.AsNoTracking().ToList());
                _cache[key] = data;
                CacheUpdated?.Invoke(this, new CacheUpdatedEventArgs(key));
            }
        }
        // 新增其他 key 的載入邏輯
    }

    public void Clear(string key) => _cache.TryRemove(key, out _);
}
```

## 程式碼：PollingService（背景輪詢、非同步、去抖）

說明：PollingService 使用 CancellationToken 支援停止，並在版本變動時以去抖（debounce）避免頻繁刷新。

```csharp
using System;
using System.Threading;
using System.Threading.Tasks;
using System.Data.Entity;
using System.Linq;
using System.Collections.Generic;

public class PollingService : IDisposable
{
    private readonly Func<AppDbContext> _dbFactory;
    private readonly CacheManager _cacheManager;
    private CancellationTokenSource _cts;
    private readonly TimeSpan _pollInterval;
    private readonly TimeSpan _debounceWindow;

    // 紀錄上次版本號
    private readonly Dictionary<string, long> _knownVersions = new Dictionary<string, long>();

    public PollingService(Func<AppDbContext> dbFactory, CacheManager cacheManager, TimeSpan pollInterval, TimeSpan debounceWindow)
    {
        _dbFactory = dbFactory;
        _cacheManager = cacheManager;
        _pollInterval = pollInterval;
        _debounceWindow = debounceWindow;
    }

    public void Start()
    {
        if (_cts != null) return;
        _cts = new CancellationTokenSource();
        Task.Run(() => PollLoopAsync(_cts.Token));
    }

    public void Stop()
    {
        _cts?.Cancel();
        _cts = null;
    }

    private async Task PollLoopAsync(CancellationToken token)
    {
        while (!token.IsCancellationRequested)
        {
            try
            {
                using (var db = _dbFactory())
                {
                    var versions = await Task.Run(() => db.VersionEntries.AsNoTracking().ToList(), token);

                    foreach (var v in versions)
                    {
                        if (!_knownVersions.TryGetValue(v.Key, out var known) || known != v.VersionNumber)
                        {
                            _knownVersions[v.Key] = v.VersionNumber;
                            // 去抖：等待短時間內是否還有變化
                            _ = Task.Run(async () =>
                            {
                                try
                                {
                                    await Task.Delay(_debounceWindow, token);
                                    await _cacheManager.RefreshAsync(v.Key);
                                }
                                catch (TaskCanceledException) { }
                            });
                        }
                    }
                }
            }
            catch (TaskCanceledException) { break; }
            catch (Exception ex)
            {
                // 記錄錯誤（log）
            }
            try { await Task.Delay(_pollInterval, token); } catch (TaskCanceledException) { break; }
        }
    }

    public void Dispose() => Stop();
}
```

## 程式碼：Autofac 註冊範例

說明：以 Func 作 factory 注入，CacheManager 與 PollingService 註冊為 Singleton。

```csharp
using Autofac;

public class AppModule : Module
{
    protected override void Load(ContainerBuilder builder)
    {
        builder.Register(c => new AppDbContext()).InstancePerDependency();
        builder.Register<Func<AppDbContext>>(c =>
        {
            var ctx = c.Resolve<IComponentContext>();
            return () => ctx.Resolve<AppDbContext>();
        });

        builder.RegisterType<CacheManager>().AsSelf().SingleInstance();
        builder.RegisterType<PollingService>().AsSelf().SingleInstance();
        builder.RegisterType<ProductRepository>().AsSelf().InstancePerDependency();
    }
}
```

## 注意事項與最佳實務

1. 輪詢頻率：SQLite 下建議較長間隔（例如 10~30 秒），避免過度讀寫造成鎖。若網路穩定、需求即時性高，可降至 3~5 秒但須測試。
2. 部分刷新：設計 VersionTable 的 Key 粒度（例如 "Products"、"Orders:BranchA"）以避免全部清空。
3. 去抖與合併：當大量快速更新時，去抖（debounce）可將多次更新合併成一次刷新。
4. 鎖與寫入衝突：SQLite 寫入會鎖整個 DB，應儘量縮短交易時間，避免長時間 transaction。
5. 日誌與錯誤處理：在 PollingService 捕獲例外並記錄，避免背景執行崩潰。
6. 安全性：在跨廠房網路上，若使用 TCP/HTTP 或 MQ，請使用加密與認證。
7. 未來可換 SQL Server：因不依賴 SqlDependency，本架構可無痛切換至 SQL Server；切換後可考慮改用 SqlDependency 或 SignalR 中介服務以降低延遲與輪詢成本。
8. 測試建議：在每個廠房部署模擬多台客戶端與模擬高頻寫入，測試 DB 鎖、輪詢負載、UI 更新頻率。

# 桌面程式分佈式 Cache 通知設計（SQLite / 內網環境）

## 1. 問題分析

1. 現在使用 SQLite
    - SQLite 是單檔資料庫，不適合多台電腦同時寫入
    - 讀取多台程式通常沒問題，但寫入容易衝突
2. 分布式桌面程式，跨辦公室及廠房
    - 有內網，但沒有 ASP / Web 服務
    - 未來可能換 SQL Server，需要彈性設計
3. 需求：資料庫變動時，通知所有程式更新快取

## 2. 可行策略（不依賴 ASP / Web）

1. **共享 SQLite + 輪詢通知表**
    
    - 在 SQLite 建一張「變動通知表」
    - 每台程式定時輪詢通知表，檢查最新版本或時間戳
    - 若有變動，更新本地 Cache
    - 優點：簡單、跨 SQLite/SQL Server 可移植
    - 缺點：非即時，輪詢頻率影響延遲與效能
        
2. **內網 P2P / Socket 廣播**
    
    - 每台桌面程式啟動一個小型 TCP/UDP Server 或使用 UDP 廣播
    - 資料變動時，程式向內網廣播「資料已更新」
    - 所有程式收到訊息後刷新 Cache
    - 優點：即時、無需中央 Server
    - 缺點：實作 Socket 較複雜，跨子網可能受限
        
3. **內網訊息中介（如 RabbitMQ / ZeroMQ）**
    
    - 架一個輕量訊息伺服器，桌面程式訂閱訊息
    - 資料變動時發送事件
    - 優點：即時、可靠，可換資料庫時不需改程式邏輯
    - 缺點：需要維護訊息服務

## 3. SQLite 特性注意

1. 單檔寫入會鎖檔，建議：
    
    - 大量寫入集中在一台電腦（Master）
    - 其他程式只讀，或透過輪詢更新
        
2. 若將來換 SQL Server，可用 SqlDependency / SignalR / Pub-Sub
3. 跨 SQLite 實作即時通知較困難，建議用 **輪詢 + 廣播訊息結合**

## 4. 建議方案（適合目前環境）

1. 建立「資料版本表」
    - 例如 `TableVersion`：`TableName`、`LastUpdateTime`、`VersionNo`
2. 每台程式維護本地快取 + 本地記錄的版本號
3. 定時輪詢 SQLite 的 `TableVersion`（例如每 2 秒）
4. 如果發現 `VersionNo` 變化，更新 Cache
5. 可選：搭配 UDP 廣播，資料更新後立即通知內網其他程式，減少輪詢延遲

## 5. 範例流程（輪詢 + UDP 廣播）

```csharp
// 定時輪詢 TableVersion
Timer timer = new Timer();
timer.Interval = 2000; // 2 秒
timer.Tick += (s,e) =>
{
    var newVersion = GetTableVersion("MyTable"); // 從 SQLite 讀取
    if(newVersion > localVersion)
    {
        UpdateCache(); 
        localVersion = newVersion;
    }
};
timer.Start();

// 廣播資料更新（可選）
UdpClient udp = new UdpClient();
byte[] message = Encoding.UTF8.GetBytes("MyTableUpdated");
udp.Send(message, message.Length, "255.255.255.255", 12345); // 廣播到內網
```

## 6. 設計建議

1. **兼容未來 SQL Server**
    - TableVersion 概念不變，只需改查詢方式
    - 若 SQL Server，可再加 SqlDependency 即時通知
2. **Cache 更新策略**
    - 盡量只更新變動資料
    - 用版本號或 timestamp 避免重複更新
3. **UDP 廣播注意**
    - 子網內可用，跨子網需配置路由
    - 若內網安全策略嚴格，可改 TCP 連線或訊息伺服器
