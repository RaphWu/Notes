# Registration Concepts 註冊概念

You register [components](https://autofac.readthedocs.io/en/latest/glossary.html) with Autofac by creating a `ContainerBuilder` and informing the builder which [components](https://autofac.readthedocs.io/en/latest/glossary.html) expose which [services](https://autofac.readthedocs.io/en/latest/glossary.html).
您可以使用 Autofac 註冊元件，方法是在 `ContainerBuilder` 中建立一個，並告訴建構器哪些元件提供哪些服務。

**Components** can be created via **reflection** (by registering a specific .NET type or open generic); by providing a ready-made **instance** (an instance of an object you created); or via lambda **expression** (an anonymous function that executes to instantiate your object). `ContainerBuilder` has a family of `Register()` methods that allow you to set these up.
**元件**可以透過**反射**（通過註冊特定的.NET 型別或開放泛型）建立；提供現成的實例（您所創建的物件實例）；或透過 lambda 表達式（一個匿名函數，當它執行時會建立您的物件）。 `ContainerBuilder` 有一系列 `Register()` 方法，讓您設定這些元件。

Each component exposes one or more **services** that are wired up using the `As()` methods on `ContainerBuilder`.
每個元件都會暴露一個或多個服務，這些服務會使用 `ContainerBuilder` 上的 `As()` 方法進行連接。

```csharp
// 建立用於註冊元件/服務的建構器。
var builder = new ContainerBuilder();

// 註冊暴露介面的類型…
builder.RegisterType<ConsoleLogger>().As<ILogger>();

// 註冊您建立的物件實例…
var output = new StringWriter();
builder.RegisterInstance(output).As<TextWriter>();

// 註冊用於建立物件的表達式…
builder.Register(c => new ConfigReader("mysection")).As<IConfigReader>();

// 建置容器以完成註冊並準備進行物件解析。
var container = builder.Build();

// 現在您可以使用 Autofac 解析服務。例如，此行將執行註冊至 IConfigReader 服務的 lambda 表達式。
using(var scope = container.BeginLifetimeScope())
{
	var reader = scope.Resolve<IConfigReader>();
}
```

## Reflection Components 反射組件

### Register by Type 按類型註冊

Components generated by reflection are typically registered by type:
由反射生成的元件通常會按類型進行註冊：

```csharp
var builder = new ContainerBuilder();
builder.RegisterType<ConsoleLogger>();
builder.RegisterType(typeof(ConfigReader));
```

When using reflection-based components, **Autofac automatically uses the constructor for your class with the most parameters that are able to be obtained from the container**.
當使用基於反射的元件時，**Autofac 會自動使用您的類別中參數最多的建構函數，並從容器中獲取這些參數。**

For example, say you have a class with three constructors like this:
例如，假設你有一個有三個建構函數的類別，如下：

```csharp
public class MyComponent
{
    public MyComponent() { /* ... */ }
    public MyComponent(ILogger logger) { /* ... */ }
    public MyComponent(ILogger logger, IConfigReader reader) { /* ... */ }
}
```

Now say you register components and services in your container like this:
現在假設你在容器中註冊元件和服務，如下：

```csharp
var builder = new ContainerBuilder();
builder.RegisterType<MyComponent>();
builder.RegisterType<ConsoleLogger>().As<ILogger>();
var container = builder.Build();

using(var scope = container.BeginLifetimeScope())
{
  var component = scope.Resolve<MyComponent>();
}
```

When you resolve your component, Autofac will see that you have an `ILogger` registered, but you don’t have an `IConfigReader` registered. In that case, the second constructor will be chosen since that’s the one with the most parameters that can be found in the container.
當你解析你的元件時，Autofac 會看到你已註冊了一個 `ILogger` ，但你沒有註冊 `IConfigReader` 。在那種情況下，將選擇第二個建構函數，因為那個建構函數有最多的參數可以在容器中找到。

**An important note on reflection-based components:** Any component type you register via `RegisterType` must be a **concrete type**. While components can expose abstract classes or interfaces as [services](https://autofac.readthedocs.io/en/latest/glossary.html), you can’t register an abstract/interface component. It makes sense if you think about it: behind the scenes, Autofac is creating an instance of the thing you’re registering. You can’t “new up” an abstract class or an interface. You have to have an implementation, right?
**關於基於反射的元件的重要注意事項**：你通過 `RegisterType` 註冊的任何元件類型必須是**實體類型**。雖然元件可以暴露抽象類別或介面作為服務，但你不能註冊抽象/介面元件。如果你這麼想，這是有道理的：在背後，Autofac 正在為你註冊的事物創建一個實例。你不能 “new up” 一個抽象類別或介面。對吧？你必須有一個實例，對吧？

### Specifying a Constructor 指定建構子

**You can manually choose a particular constructor** to use and override the automatic choice by registering your component with the `UsingConstructor` method and a list of types representing the parameter types in the constructor:
您可以手動選擇特定的建構子來使用，並透過使用 `UsingConstructor` 方法及建構子中參數類型的型別列表註冊您的元件來覆蓋自動選擇：

```csharp
builder.RegisterType<MyComponent>()
       .UsingConstructor(typeof(ILogger), typeof(IConfigReader));
```

Note that you will still need to have the requisite parameters available at resolution time or there will be an error when you try to resolve the object. You can [pass parameters at registration time](https://autofac.readthedocs.io/en/latest/register/parameters.html) or you can [pass them at resolve time](https://autofac.readthedocs.io/en/latest/resolve/parameters.html).
注意，您仍需要在解析度時提供必要的參數，否則在嘗試解析物件時會出現錯誤。您可以在註冊時傳遞參數，也可以在解析時傳遞。

> Note  注意
> You can find advanced methods of customising which constructor to use [here](https://autofac.readthedocs.io/en/latest/advanced/constructor-selection.html).
> 您可以在這裡找到自訂使用哪個建構函數的進階方法。

### Required Properties 所需屬性

Starting in Autofac 7.0, in a reflection-based component, all [required properties](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/keywords/required) are automatically resolved, in the same manner as constructor parameters.
從 Autofac 7.0 開始，在基於反射的元件中，所有 [必要的屬性](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/keywords/required) 都會自動解析，方式與建構子參數相同。

All required properties of the component _must_ be resolvable services (or supplied as a [parameter](https://autofac.readthedocs.io/en/latest/resolve/parameters.html)) otherwise an exception will be thrown when trying to resolve the component.
元件的所有必要屬性 *必須* 是可解析的服務（或作為 [參數](https://autofac.readthedocs.io/en/latest/resolve/parameters.html) 提供），否則在嘗試解析元件時將引發異常。

For example, consider a class with these properties:
例如，考慮具有以下屬性的類別：

```csharp
public class MyComponent
{
  public required ILogger Logger { protected get; init; }

  public required IConfigReader ConfigReader { protected get; init; }
}
```

You can register and use this class as you could if it had a constructor:
您可以像有建構函數一樣註冊和使用這個類別：

```csharp
var builder = new ContainerBuilder();
builder.RegisterType<MyComponent>();
builder.RegisterType<ConsoleLogger>().As<ILogger>();
builder.RegisterType<ConfigReader>().As<IConfigReader>();
var container = builder.Build();

using(var scope = container.BeginLifetimeScope())
{
  // Logger and ConfigReader will be populated. 被填充
  var component = scope.Resolve<MyComponent>();
}
```

Required properties are also set automatically on all base classes (if they are present); this makes required properties useful with deep object hierarchies, because it allows you to avoid having to invoke base constructors with the set of services; Autofac will set the base class properties for you.
必要的屬性也會自動設定在所有基礎類別上（如果存在）；這使得必要的屬性在深層物件階層中非常有用，因為它允許您避免必須使用服務集合調用基礎建構函數；Autofac 會為您設定基礎類別的屬性。

> Note  注意
> For more details on required property injection, see the dedicated section in the [property injection documentation](https://autofac.readthedocs.io/en/latest/register/prop-method-injection.html).
> 有關必要屬性注入的更多詳細資訊，請參閱屬性注入文檔中的專屬部分。

## Instance Components 實例組件 

In some cases, you may want to pre-generate an instance of an object and add it to the container for use by registered components. You can do this using the `RegisterInstance` method:
在某些情況下，你可能需要預先產生一個物件實例，並將其新增至容器中，以供已註冊的元件使用。你可以使用 `RegisterInstance` 方法來實現這一點：

```csharp
var output = new StringWriter();
builder.RegisterInstance(output).As<TextWriter>();
```

Something to consider when you do this is that Autofac [automatically handles disposal of registered components](https://autofac.readthedocs.io/en/latest/lifetime/disposal.html) and you may want to control the lifetime yourself rather than having Autofac call `Dispose` on your object for you. In that case, you need to register the instance with the `ExternallyOwned` method:
執行此操作時需要考慮的一點是，Autofac [會自動處理已註冊元件的釋放](https://autofac.readthedocs.io/en/latest/lifetime/disposal.html) ，您可能希望自己控制生命週期，而不是讓 Autofac 為您呼叫物件的 `Dispose` 方法。在這種情況下，您需要使用 `ExternallyOwned` 方法註冊實例：

```csharp
var output = new StringWriter();
builder.RegisterInstance(output)
       .As<TextWriter>()
       .ExternallyOwned();
```

Registering provided instances is also handy when integrating Autofac into an existing application where a singleton instance already exists and needs to be used by components in the container. Rather than tying those components directly to the singleton, it can be registered with the container as an instance:
當 Autofac 整合到現有的應用程式中時，如果已經存在單例實例，並且需要由容器中的元件使用，那麼註冊提供的實例也很方便。與其將這些元件直接綁定到單例，不如將其作為實例註冊到容器中：

```csharp
builder.RegisterInstance(MySingleton.Instance).ExternallyOwned();
```

This ensures that the static singleton can eventually be eliminated and replaced with a container-managed one.
這確保了靜態單例最終可以被消除並被容器管理的單例所取代。

The default service exposed by an instance is the concrete type of the instance. See “Services vs. Components,” below.
實例公開的預設服務是該實例的具體類型。請參閱下文的「服務與組件」。

## Lambda Expression Components Lambda 表達式元件 

Reflection is a pretty good default choice for component creation. Things get messy, though, when component creation logic goes beyond a simple constructor call.
反射是組件創建的一個很好的預設選擇。然而，當元件創建邏輯超出簡單的建構函數呼叫時，事情就會變得混亂。

Autofac can accept a delegate or lambda expression to be used as a component creator:
Autofac 可以接受委託或 lambda 表達式作為元件建立器：

```csharp
builder.Register(c => new A(c.Resolve<B>()));
```

The parameter `c` provided to the expression is the _component context_ (an `IComponentContext` object) in which the component is being created. You can use this to resolve other values from the container to assist in creating your component. **It is important to use this rather than a closure to access the container** so that [deterministic disposal](https://autofac.readthedocs.io/en/latest/lifetime/disposal.html) and nested containers can be supported correctly.
傳遞給表達式的參數 `c` 是 *元件上下文* （一個 `IComponentContext` 物件），元件將在其中建立。您可以使用它來解析容器中的其他值，以協助建立元件。 **務必使用這個參數而不是閉包來存取容器，** 這樣才能正確支援 [確定性處置](https://autofac.readthedocs.io/en/latest/lifetime/disposal.html) 和嵌套容器。

Additional dependencies can be satisfied using this context parameter - in the example, `A` requires a constructor parameter of type `B` that may have additional dependencies.
可以使用此上下文參數來滿足額外的依賴關係 - 在範例中， `A` 需要類型 `B` 的建構函數參數，該參數可能具有額外的依賴關係。

As well as using `IComponentContext` to resolve dependencies in your lambda expression, you can also use the generic `Register` overloads to specify your dependencies as a variable number of typed arguments to the lambda, and Autofac will resolve them for you:
除了使用 `IComponentContext` 解析 lambda 表達式中的依賴項之外，您還可以使用通用 `Register` 重載將相依性指定為 lambda 的可變數量的型別參數，Autofac 將為您解析它們：

```csharp
builder.Register((IDependency1 dep1, IDependency2 dep2) => new Component(dep1, dep2));
```

You can blend the `IComponentContext` and generic approach if you need to make conditional choices, or use methods like `ResolveNamed`. Just add the `IComponentContext` as the first parameter to the lambda:
如果需要進行條件選擇，可以混合使用 `IComponentContext` 和通用方法，或使用 `ResolveNamed` 之類的方法。只需將 `IComponentContext` 作為 lambda 的第一個參數添加即可：

```csharp
builder.Register((IComponentContext ctxt, IDependency1 dep1)
	=> new Component(dep1, ctxt.ResolveNamed<IDependency2>("value")));
```

The default service provided by an expression-created component is the inferred return type of the expression.
由表達式建立的元件提供的預設服務是表達式的推斷回傳類型。

Below are some examples of requirements met poorly by reflective component creation but nicely addressed by lambda expressions.
以下是一些透過反射元件創建無法很好地滿足需求但透過 lambda 表達式可以很好地解決的需求範例。

### Complex Parameters 複雜參數

Constructor parameters can’t always be declared with simple constant values. Rather than puzzling over how to construct a value of a certain type using an XML configuration syntax, use code:
建構函數參數並不總是可以用簡單的常數值來宣告。與其費解如何使用 XML 配置語法建構特定類型的值，不如使用以下程式碼：

```csharp
builder.Register(c => new UserSession(DateTime.Now.AddMinutes(25)));
```

(Of course, session expiry is probably something you’d want to specify in a configuration file - but you get the gist ;))
（當然，會話到期可能是您想要在設定檔中指定的 - 但您明白要點；））

### Property Injection 屬性注入

While Autofac offers [a more first-class approach to property injection](https://autofac.readthedocs.io/en/latest/register/prop-method-injection.html), you can use ***expressions*** and **property initializers** to populate properties as well:
雖然 Autofac 提供了 [更一流的屬性注入方法](https://autofac.readthedocs.io/en/latest/register/prop-method-injection.html) ，但您也可以使用**表達式**和**屬性初始化器**來填充屬性：

```csharp
builder.Register(c => new A() { MyB = c.ResolveOptional<B>() });
```

The `ResolveOptional` method will try to resolve the value but won’t throw an exception if the service isn’t registered. (You will still get an exception if the service is registered but can’t properly be resolved.) This is one of the options for [resolving a service](https://autofac.readthedocs.io/en/latest/resolve/index.html).
`ResolveOptional` 方法會嘗試解析該值，但如果服務未註冊則不會拋出例外（如果服務已註冊但無法正確解析，您仍然會收到例外）。這是 [解析服務](https://autofac.readthedocs.io/en/latest/resolve/index.html) 的選項之一。

**Property injection is not recommended in the majority of cases.** Alternatives like [the Null Object pattern](http://en.wikipedia.org/wiki/Null_Object_pattern), overloaded constructors or constructor parameter default values make it possible to create cleaner, “immutable” components with optional dependencies using constructor injection.
**大多數情況下不建議使用屬性注入。** [空物件模式](http://en.wikipedia.org/wiki/Null_Object_pattern) 、重載建構函式或建構函式參數預設值等替代方案，可以透過**建構函式註入**來建立更簡潔、具有可選依賴項的「不可變」元件。

### Selection of an Implementation by Parameter Value 透過參數值選擇實現

One of the great benefits of isolating component creation is that the concrete type can be varied. This is often done at runtime, not just configuration time:
將元件建立隔離的一大好處是，具體型別可以變化。這通常是在 runtime 而非僅僅是 configuration time 進行：

```csharp
builder.Register<CreditCard>((c, p) =>
{
    var accountId = p.Named<string>("accountId");
    if (accountId.StartsWith("9"))
    {
        return new GoldCard(accountId);
    }
    else
    {
        return new StandardCard(accountId);
    }
});
```

In this example, `CreditCard` is implemented by two classes, `GoldCard` and `StandardCard` - which class is instantiated depends on the account ID provided at runtime.
在這個例子中， `CreditCard` 由兩個類別 `GoldCard` 和 `StandardCard` 實作 - 實例化哪個類別取決於執行時提供的帳戶 ID。

[Parameters are provided to the creation function](https://autofac.readthedocs.io/en/latest/resolve/parameters.html) through an optional second parameter named `p` in this example.
在此範例中，透過名為 `p` 的可選第二個參數 [向建立函數提供參數](https://autofac.readthedocs.io/en/latest/resolve/parameters.html) 。

Using this registration would look like:
使用此註冊將如下所示：

```csharp
var card = container.Resolve<CreditCard>(new NamedParameter("accountId", "12345"));
```

You can use parameters with the `Func<X, Y>` [relationship](https://autofac.readthedocs.io/en/latest/resolve/relationships.html) by using typed instead of named parameters.
您可以使用類型參數而不是命名參數來使用具有 `Func<X, Y>` [關係](https://autofac.readthedocs.io/en/latest/resolve/relationships.html) 的參數。

```csharp
builder.Register<CreditCard>((c, p) =>
{
    // Typed instead of named to work with Func<X, Y>
    var accountId = p.TypedAs<string>();
    if (accountId.StartsWith("9"))
    {
        return new GoldCard(accountId);
    }
    else
    {
        return new StandardCard(accountId);
    }
});
```

Using this registration would look like:
使用此註冊將如下所示：

```csharp
var cardFactory = container.Resolve<Func<string, CreditCard>>();
var card = cardFactory("12345");
```

You can use [lambda expression components](https://autofac.readthedocs.io/en/latest/register/registration.html#register-registration-lambda-expression-components) here, too, to make it even simpler:
您也可以在這裡使用 [lambda 表達式元件](https://autofac.readthedocs.io/en/latest/register/registration.html#register-registration-lambda-expression-components) ，使其更簡單：

```csharp
builder.Register<string, CreditCard>((accountId) =>
{
    if (accountId.StartsWith("9"))
    {
        return new GoldCard(accountId);
    }
    else
    {
        return new StandardCard(accountId);
    }
});
```

Another clean, type-safe syntax can be achieved if a delegate to create `CreditCard` instances is declared and [a delegate factory](https://autofac.readthedocs.io/en/latest/advanced/delegate-factories.html) is used. Delegate factories are the way you can support more than one parameter of the same type.
如果聲明一個委託來建立 `CreditCard` 實例，並使用 [委託工廠](https://autofac.readthedocs.io/en/latest/advanced/delegate-factories.html) ，則可以實作另一種簡潔、型別安全的語法。委託工廠可以支援同一類型的多個參數。

## Open Generic Components 開放式通用組件

Autofac supports open generic types. Use the `RegisterGeneric()` builder method:
Autofac 支援開放式通用型別。使用 `RegisterGeneric()` 建構器方法：

```csharp
builder.RegisterGeneric(typeof(NHibernateRepository<>))
       .As(typeof(IRepository<>))
       .InstancePerLifetimeScope();
```

When a matching service type is requested from the container, Autofac will map this to an equivalent closed version of the implementation type:
當從容器請求符合的服務型別時，Autofac 會將其映射至實現型別的等效封閉版本：

```csharp
// Autofac will return an NHibernateRepository<Task>
var tasks = container.Resolve<IRepository<Task>>();
```

Registration of a specialized service type (e.g. `IRepository<Person>`) will override the open generic version.
專門服務類型（例如 `IRepository<Person>` ）的註冊會覆蓋開放的泛型版本。

You can also use delegates to provide the closed generic type, if you have some custom behavior for choosing the closed generic implementation:
如果您有一些自訂行為來選擇封閉泛型實作，也可以使用委託來提供封閉泛型類型：

```csharp
var builder = new ContainerBuilder();

builder.RegisterGeneric((context, types, parameters) =>
{
    // Make decisions about what closed type to use.
    if (types.Contains(typeof(string)))
    {
        return new StringSpecializedImplementation();
    }

    return Activator.CreateInstance(typeof(GeneralImplementation<>).MakeGenericType(types));
}).As(typeof(IService<>));
```

> Note  注意
>
> Be aware that the delegate form of `RegisterGeneric` will usually be **slightly less performant** than the reflection-based version, because the closed generic type cannot be cached in the same way.
> 請注意， `RegisterGeneric` 的委託形式通常會比基於反射的版本**稍微效能較低**，因為封閉泛型類型無法以相同的方式進行快取。

## Services vs. Components 服務與元件

When you register [components](https://autofac.readthedocs.io/en/latest/glossary.html), you have to tell Autofac which [services](https://autofac.readthedocs.io/en/latest/glossary.html) that component exposes. By default, most registrations will just expose themselves as the type registered:
當你註冊元件時，你必須告訴 Autofac 這個元件會公開哪些服務。預設情況下，大多數註冊都只會以註冊的類型公開自己：

```csharp
// This exposes the service "CallLogger"
builder.RegisterType<CallLogger>();
```

Components can only be [resolved](https://autofac.readthedocs.io/en/latest/resolve/index.html) by the services they expose. In this simple example it means:
元件只能透過它們公開的服務來解析。在這個簡單的範例中，這表示：

```csharp
// This will work because the component exposes the type by default:
// 這樣做是可行的，因為該元件預設會暴露類型：
scope.Resolve<CallLogger>();

// This will NOT work because we didn't tell the registration to also expose the ILogger interface on CallLogger:
// 這將行不通，因為我們沒有告訴註冊過程也要在 CallLogger 上公開 ILogger 介面：
scope.Resolve<ILogger>();
```

You can expose a component with any number of services you like:
你可以公開任意數量的服務：

```csharp
builder.RegisterType<CallLogger>()
       .As<ILogger>()
       .As<ICallInterceptor>();
```

Once you expose a service, you can resolve the component based on that service. Note, however, that once you expose a component as a specific service, the default service (the component type) is overridden:
一旦您公開一個服務，您就可以根據這個服務解析元件。然而，請注意，一旦您將元件公開為特定的服務，預設服務（元件類型）就會被覆蓋：

```csharp
// These will both work because we exposed the appropriate services in the registration:
// 這兩種方法都能正常運作，因為我們在註冊時暴露了相應的服務：
scope.Resolve<ILogger>();
scope.Resolve<ICallInterceptor>();

// This WON'T WORK anymore because we specified service overrides on the component:
// 這種方法不再有效，因為我們在元件中指定了服務覆寫：
scope.Resolve<CallLogger>();
```

If you want to expose a component as a set of services as well as using the default service, use the `AsSelf` method:
如果您想將元件作為一組服務公開，同時也使用預設服務，請使用 `AsSelf` 方法：

```csharp
builder.RegisterType<CallLogger>()
       .AsSelf()
       .As<ILogger>()
       .As<ICallInterceptor>();
```

Now all of these will work:
現在這些都會正常工作：

```csharp
// These will all work because we exposed the appropriate services in the registration:
// 這些功能都能正常運作，因為我們在註冊過程中啟用了相應的服務：
scope.Resolve<ILogger>();
scope.Resolve<ICallInterceptor>();
scope.Resolve<CallLogger>();
```

## Default Registrations 預設註冊

If more than one component exposes the same service, **Autofac will use the last registered component as the default provider of that service**:
若有多個元件公開相同服務，Autofac 會使用**最後註冊的元件**作為該服務的預設提供者：

```csharp
builder.RegisterType<ConsoleLogger>().As<ILogger>();
builder.RegisterType<FileLogger>().As<ILogger>();
```

In this scenario, `FileLogger` will be the default for `ILogger` because it was the last one registered.
在此情況下， `FileLogger` 將是 `ILogger` 的預設值，因為它是最後註冊的。

To override this behavior, use the `PreserveExistingDefaults()` modifier:
若要覆蓋此行為，請使用 `PreserveExistingDefaults()` 修飾詞：

```csharp
builder.RegisterType<ConsoleLogger>().As<ILogger>();
builder.RegisterType<FileLogger>().As<ILogger>().PreserveExistingDefaults();
```

In this scenario, `ConsoleLogger` will be the default for `ILogger` because the later registration for `FileLogger` used `PreserveExistingDefaults()`.
在此情況下， `ConsoleLogger` 將是 `ILogger` 的預設值，因為 `FileLogger` 的後續註冊使用了 `PreserveExistingDefaults()` 。

## Conditional Registration 條件式註冊

> Note  注意
>
> Conditional registration was introduced in Autofac **4.4.0**
> 條件式註冊是在 Autofac 4.4.0 中引進的

In most cases, overriding registrations as noted in the section above, “Default Registrations,” is enough to get the right component resolved at runtime. Ensuring things get registered in the right order; using `PreserveExistingDefaults()`; and taking advantage of lambda/delegate registrations for more complex conditions and behavior can get you pretty far.
在大多數情況下，覆寫註冊如上「預設註冊」部分所述，就足以在執行時間解析到正確的元件。確保事物以正確的順序註冊；使用 `PreserveExistingDefaults()` ；並利用 lambda/委託註冊來處理更複雜的條件與行為，可以讓你走得很遠。

There can be a few scenarios where this may not be the way you want to go:
可能存在幾種情況，這可能不是您想要的處理方式：

- You don’t want the component present in the system if something else is handling the functionality. For example, if you resolve an `IEnumerable<T>` of a service, all of the registered components implementing that service will be returned, whether or not you’ve used `PreserveExistingDefaults()`. Usually this is fine, but there are some edge cases where you may not want that.
  您不希望元件出現在系統中，如果其他東西正在處理該功能。例如，如果您解析服務的 `IEnumerable<T>` ，所有註冊的實現該服務的元件都將被返回，無論您是否使用過 `PreserveExistingDefaults()` 。通常這沒有問題，但在某些邊界情況下，您可能不希望這樣。

- You only want to register the component if some other component _isn’t_ registered; or only if some other component _is_ registered. You can’t resolve things out of a container that you’re building, and you shouldn’t update a container that’s already built. Being able to conditionally register a component based on other registrations can be helpful.
  您只希望在未註冊其他元件時才註冊該元件；或者只有在已註冊其他元件時才註冊。您無法從正在建立的容器中解析物件，也不應更新已建立的容器。能夠根據其他註冊情況條件性註冊元件可能會很有幫助。

There are two registration extensions that can help in these cases:
有兩個註冊擴充可以幫助處理這些情況：

- `OnlyIf()` - Provide a lambda that uses an `IComponentRegistryBuilder` to determine if a registration should happen.
  `OnlyIf()` - 提供一個 lambda，使用 `IComponentRegistryBuilder` 來判斷是否應該發生註冊。
    
- `IfNotRegistered()` - Shortcut to stop a registration from happening if some other service is already registered.
  `IfNotRegistered()` - 簡短方式，如果其他服務已經註冊，則停止註冊發生。

These extensions run at the time of `ContainerBuilder.Build()` and will execute in the order of the actual component registrations. Here are some examples showing how they work:
這些擴充功能在 `ContainerBuilder.Build()` 時執行，並將按照實際元件註冊的順序執行。這裡有一些示例說明它們如何工作：

```csharp
var builder = new ContainerBuilder();

// Only ServiceA will be registered.
// Note the IfNotRegistered takes the SERVICE TYPE to check for (the As<T>), NOT the COMPONENT TYPE (the RegisterType<T>).
// 注意 IfNotRegistered 接受的是要檢查的服務類型（As<T>），而不是元件類型（RegisterType<T>）。
builder.RegisterType<ServiceA>()
       .As<IService>();
builder.RegisterType<ServiceB>()
       .As<IService>()
       .IfNotRegistered(typeof(IService));

// HandlerA WILL be registered - it's running BEFORE HandlerB has a chance to be registered so the IfNotRegistered check won't find it.
//
// HandlerC will NOT be registered because it runs AFTER HandlerB. Note it can check for the type "HandlerB" because HandlerB registered AsSelf() not just As<IHandler>(). Again, IfNotRegistered can only check for "As" types.
//
// HandlerA 將會註冊——它在 HandlerB 註冊之前運行，因此 IfNotRegistered 檢查不會發現它。
//
// HandlerC 不會註冊，因為它在 HandlerB 之後運行。注意，它可以檢查類型“HandlerB”，因為 HandlerB 註冊的是 AsSelf() 而不是 As<IHandler>()。同樣，IfNotRegistered 只能檢查「As」類型。
builder.RegisterType<HandlerA>()
       .AsSelf()
       .As<IHandler>()
       .IfNotRegistered(typeof(HandlerB));
builder.RegisterType<HandlerB>()
       .AsSelf()
       .As<IHandler>();
builder.RegisterType<HandlerC>()
       .AsSelf()
       .As<IHandler>()
       .IfNotRegistered(typeof(HandlerB));

// Manager will be registered because both an IService and HandlerB are registered. The OnlyIf predicate can allow a lot more flexibility.
// 由於 IService 和 HandlerB 都已註冊，因此 Manager 也將註冊。 OnlyIf 謂詞可以提供更大的彈性。
builder.RegisterType<Manager>()
       .As<IManager>()
       .OnlyIf(reg =>
            reg.IsRegistered(new TypedService(typeof(IService))) &&
            reg.IsRegistered(new TypedService(typeof(HandlerB))));

// Be careful with open generics - IfNotRegistered and IsRegistered only work with CLOSED generics because that's what you'd resolve!
// Note the check is for a closed generic that would be registered as part of the open generic - if you put an open generic in the IfNotRegistered check, it'll always appear not registered.
// 注意開放泛型－IfNotRegistered 和 IsRegistered 只適用於封閉泛型，因為這是你需要解析的物件！
// 注意，此檢查針對的是作為開放泛型一部分註冊的封閉泛型——如果你將開放泛型放入 IfNotRegistered 檢查中，它將始終顯示為未註冊。
builder.RegisterGeneric(typeof(CommandHandler<>))
       .As(typeof(ICommandHandler<>))
       .IfNotRegistered(typeof(ICommandHandler<MyCommand>));

// This is when the conditionals actually run. Again, they run in the order the registrations were added to the ContainerBuilder.
// 這是條件語句實際執行的時候。同樣，它們的執行順序與註冊資訊加入 ContainerBuilder 的順序相同。
var container = builder.Build();
```

## Configuration of Registrations 註冊設定

You can [use XML or programmatic configuration (“modules”)](https://autofac.readthedocs.io/en/latest/configuration/index.html) to provide groups of registrations together or change registrations at runtime. You can also use [use Autofac modules](https://autofac.readthedocs.io/en/latest/configuration/modules.html) for some dynamic registration generation or conditional registration logic.
您可以使用 XML 或程式性設定（「模組」）來一起提供一群註冊或於執行時間變更註冊。您也可以使用 Autofac 模組來進行某些動態註冊產生或條件註冊邏輯。

## Dynamically-Provided Registrations 動態提供的註冊

[Autofac modules](https://autofac.readthedocs.io/en/latest/configuration/modules.html) are the simplest way to introduce dynamic registration logic or simple cross-cutting features. For example, you can use a module to [dynamically attach a log4net logger instance to a service being resolved](https://autofac.readthedocs.io/en/latest/examples/log4net.html).
Autofac 模組是引入動態註冊邏輯或簡單跨切面功能的最簡單方式。例如，您可以使用模組來動態地將一個 log4net 記錄器實例附加到正在解析的服務。

If you find that you need even more dynamic behavior, such as adding support for a new [implicit relationship type](https://autofac.readthedocs.io/en/latest/resolve/relationships.html), you might want to [check out the registration sources section in the advanced concepts area](https://autofac.readthedocs.io/en/latest/advanced/registration-sources.html).
如果您發現您需要更多動態行為，例如添加對新的隱含關係類型的支援，您可能想要檢視先進概念區域中的註冊來源部分。

# Passing Parameters to Register 傳遞參數至註冊

When you [register components](https://autofac.readthedocs.io/en/latest/register/registration.html) you have the ability to provide a set of parameters that can be used during the [resolution of services](https://autofac.readthedocs.io/en/latest/resolve/index.html) based on that component. (If you’d rather provide the parameters at resolution time, [you can do that instead](https://autofac.readthedocs.io/en/latest/resolve/parameters.html).)
當你註冊元件時，你可以提供一組參數，這些參數可以在基於該元件的服務解析期間使用。(如果你希望改在解析時間提供參數，你也可以選擇那樣做。)

## Available Parameter Types 可用的參數類型

Autofac offers several different parameter matching strategies:
Autofac 提供了多種不同的參數匹配策略：

- `NamedParameter` - match target parameters by name
  `NamedParameter` - 根據名稱匹配目標參數
    
- `TypedParameter` - match target parameters by type (exact type match required)
  `TypedParameter` - 根據類型匹配目標參數（需要精確類型匹配）
    
- `ResolvedParameter` - flexible parameter matching
  `ResolvedParameter` - 健壯的參數匹配

`NamedParameter` and `TypedParameter` can supply constant values only.
`NamedParameter` 和 `TypedParameter` 只能提供常數值。

`ResolvedParameter` can be used as a way to supply values dynamically retrieved from the container, e.g. by resolving a service by name.
`ResolvedParameter` 可以用來作為動態取得容器內值的方 式，例如透過以名稱解析服務。

## Parameters with Reflection Components 使用反射元件的參數

When you register a reflection-based component, the constructor of the type may require a parameter that can’t be resolved from the container. You can use a parameter on the registration to provide that value.
當你註冊基於反射的元件時，該類型的建構函數可能需要一個無法從容器中解析的參數。你可以使用註冊上的參數來提供該值。

Say you have a configuration reader that needs a configuration section name passed in:
假設你有一個需要傳入配置節點名稱的配置讀取器：

```csharp
public class ConfigReader : IConfigReader
{
    public ConfigReader(string configSectionName)
    {
        // Store config section name
    }

    // ...read configuration based on the section name.
}
```

You could use a lambda expression component for that:
你可以使用 lambda 表達式元件來做到這個：

```csharp
builder.Register(c => new ConfigReader("sectionName")).As<IConfigReader>();
```

Or you could pass a parameter to a reflection component registration:
或者你可以將參數傳遞給反射元件註冊：

```csharp
// Using a NAMED parameter:
builder.RegisterType<ConfigReader>()
        .As<IConfigReader>()
        .WithParameter("configSectionName", "sectionName");

// Using a TYPED parameter:
builder.RegisterType<ConfigReader>()
        .As<IConfigReader>()
        .WithParameter(TypedParameter.From("sectionName"));

// Using a RESOLVED parameter:
builder.RegisterType<ConfigReader>()
        .As<IConfigReader>()
        .WithParameter(new ResolvedParameter(
            (pi, ctx) => pi.ParameterType == typeof(string) && pi.Name == "configSectionName",
            (pi, ctx) => "sectionName")
        );
```

## Parameters with Lambda Expression Components 使用 Lambda 表達式元件的參數

With lambda expression component registrations, rather than passing the parameter value _at registration time_ you enable the ability to pass the value _at service resolution time_. ([Read more about resolving with parameters.](https://autofac.readthedocs.io/en/latest/resolve/parameters.html))
在 lambda 表達式元件註冊中，你並非在註冊時傳遞參數值，而是啟用在服務解析時傳遞值的能力。(閱讀更多有關使用參數解析的資訊。)

In the component registration expression, you can make use of the incoming parameters by using the generic lambda `Register` methods:
在元件註冊表達式中，您可以使用通用 lambda `Register` 方法來使用傳入的參數：

```csharp
builder.Register((MyConfig config) => new Worker(config));
```

If you need access to the full list of parameters, it’s available by changing the delegate signature you use for registration. Instead of specifying the parameter as an argument to the lambda, take in an `IComponentContext` and an `IEnumerable<Parameter>`:
如果您需要存取完整的參數列表，透過更改您用於註冊的委派簽章即可取得。不要將參數指定為 lambda 的參數，而是接收一個 `IComponentContext` 和一個 `IEnumerable<Parameter>` ：

```csharp
// Use TWO parameters to the registration delegate:
// c = The current IComponentContext to dynamically resolve dependencies
// p = An IEnumerable<Parameter> with the incoming parameter set
// 註冊委託使用兩個參數：
// c = 目前 IComponentContext，用於動態解析依賴項
// p = 包含傳入參數集的 IEnumerable<Parameter>
builder.Register((c, p) =>
                 new Worker(p.Named<MyConfig>("config")));
```

When [resolving with parameters](https://autofac.readthedocs.io/en/latest/resolve/parameters.html), your lambda will use the parameters passed in:
在帶參數解析時，您的 lambda 將使用傳入的參數：

```csharp
var customConfig = new MyConfig
{
    SomeValue = "../"
};

var worker = scope.Resolve<Worker>(new NamedParameter("config", customConfig));
```

# Property and Method Injection 屬性與方法注入

While constructor parameter injection is the preferred method of passing values to a component being constructed, you can also use property or method injection to provide values.
雖然建構函數參數注入是傳遞值給正在建構的元件的首選方法，但您也可以使用屬性或方法注入來提供值。

**Property injection** uses writeable properties rather than constructor parameters to perform injection. **Method injection** sets dependencies by calling a method.
屬性注入使用可寫入的屬性，而不是建構函數參數來進行注入。方法注入透過呼叫方法來設定相依性。

## Property Injection 屬性注入

If the component is a [lambda expression component](https://autofac.readthedocs.io/en/latest/register/registration.html#register-registration-lambda-expression-components), use an object initializer:
若元件為 lambda 表達式元件，請使用物件初始設定：

```csharp
builder.Register(c => new A { B = c.Resolve<B>() });
```

To support [circular dependencies](https://autofac.readthedocs.io/en/latest/advanced/circular-dependencies.html), use an [activated event handler](https://autofac.readthedocs.io/en/latest/lifetime/events.html):
為了支援迴圈相依性，請使用啟動事件處理程序：

```csharp
builder.Register(c => new A()).OnActivated(e => e.Instance.B = e.Context.Resolve<B>());
```

### Required Properties 必要屬性

From Autofac 7.0 onwards, for [reflection components](https://autofac.readthedocs.io/en/latest/register/registration.html#register-registration-reflection-components), all [required properties](https://learn.microsoft.com/en-us/dotnet/csharp/language-reference/keywords/required) are automatically resolved at the time of object construction, and are generally treated in much the same way as mandatory constructor arguments.
從 Autofac 7.0 開始，對於反射元件，所有必要的屬性在物件建立時自動解析，且通常與強制建構函式參數以相似的方式處理。

For example, given the following type:
例如，給定下列型別：

```csharp
public class MyComponent
{
    public required ILogger Logger { protected get; init; }
    
    public required IConfigReader ConfigReader { protected get; init; }
    
    public IDatabaseContext Context { get; set; }
}
```

When the component is resolved, Autofac will populate the `Logger` and `ConfigReader` properties as if they were constructor parameters. The `Context` property will be treated like a standard property and will not be populated by default.
當元件被解析時，Autofac 會將 `Logger` 和 `ConfigReader` 屬性視為建構函數參數般填入。而 `Context` 屬性則會被視為標準屬性，且預設不會被填入。

You can use any valid combination of access modifiers on `required` properties, however, `public required ... { protected get; init; }` is used in these examples as it provides access and visibility that are similar to constructors: the property is only settable at construction and not publicly visible to other classes.
您可以在 `required` 屬性上使用任何有效的存取修飾字組合，不過，在這些範例中使用了 `public required ... { protected get; init; }` ，因為它提供了類似建構函數的存取與可見性：屬性僅能在建構時設定，且對其他類別不公開可見。

`Required` property injection also works automatically in all base classes with `required` properties:
`required` 屬性注入也會自動在所有包含 `required` 屬性的基底類別中運作：

```csharp
public class ComponentBase
{
    public required ILogger Logger { protected get; init; }
}

public class MyComponent : ComponentBase
{
    public required IConfigReader ConfigReader { protected get; init; }
}
```

In the above example, resolving `MyComponent` would populate `Logger` in the base class, as well as `ConfigReader` in the component itself.
在上述範例中，解析 `MyComponent` 會在基底類別中填入 `Logger` ，同時也會在元件本身中填入 `ConfigReader` 。

> Important  重要
>
> Autofac does _not_ consider the nullability of the type of a required property to indicate any sort of “optional” required property. If the property is marked as `required`, then it is required, and must be injected, or provided via a parameter, regardless of its nullability.
> Autofac **不會**考慮必要屬性的型別的 nullability 來表示任何形式的「選擇性」必要屬性。如果屬性標示為 `required` ，則它是必要的，必須被注入，或透過參數提供，無論其 nullability。

#### Required Properties and Constructors 必要屬性與建構子

You can mix-and-match constructors and required properties if you so wish:
您可以自由組合使用建構子與必要屬性：

```csharp
public class MyComponent
{
    public MyComponent(ILogger logger)
    {
        Logger = logger;
    }
    
    private ILogger Logger { get; set; }
    
    public required IConfigReader ConfigReader { protected get; init; }
}
```

When multiple constructors are available, by default Autofac selects **the constructor with the most matching parameters** (unless [custom constructor selection is used](https://autofac.readthedocs.io/en/latest/advanced/constructor-selection.html)). This remains the case, and the set of required properties has no impact on the selected constructor.
當有多個建構子可用時，預設情況下 Autofac 會選擇**參數匹配度最高的建構子**（除非使用自訂建構子選擇）。這個行為仍然有效，且所需屬性的集合對於選擇的建構子沒有影響。

Autofac has no idea whether or not you set a given required property inside a constructor. Take this example:
Autofac 無法知道你是否在建構子中設定了某個所需屬性。這是一個例子：

```csharp
public class MyComponent
{
    public MyComponent()
    {
    }
    
    public MyComponent(ILogger logger)
    {
        Logger = logger;
    }
    
    public required ILogger Logger { protected get; init; }
}
```

Here, the constructor that Autofac will pick is going to be the one that takes the `ILogger` parameter, which in turn sets the `Logger` property. However, since `Logger` is marked as a required property, Autofac will resolve `ILogger` a second time, and inject it into the required property.
這裡，Autofac 將選擇接收 `ILogger` 參數的建構子，而這個參數會設定 `Logger` 屬性。然而，因為 `Logger` 被標記為所需屬性，Autofac 會再次解析 `ILogger` ，並將其注入到所需屬性中。

To avoid this, mark constructors that set all your required properties with the [SetsRequiredMembers](https://learn.microsoft.com/en-us/dotnet/api/system.diagnostics.codeanalysis.setsrequiredmembersattribute) attribute:
為了避免這個問題，請使用 `SetsRequiredMembers` 屬性標記設定所有所需屬性的建構子：

```csharp
using System.Diagnostics.CodeAnalysis;

public class MyComponent
{
    public MyComponent()
    {
    }
    
    [SetsRequiredMembers]
    public MyComponent(ILogger logger)
    {
        Logger = logger;
    }
    
    public required ILogger Logger { protected get; init; }
}
```

Since the constructor is marked as setting all required members, no required property injection will occur in Autofac, when _that constructor_ is used to create an instance of the component.
自構造函數被標記為設定所有必要的成員，當使用該構造函數來創建元件的實例時，Autofac 中不會發生必要的屬性注入。

#### Required Properties and Parameters 必要的屬性與參數

Any `TypedParameter` provided at [registration](https://autofac.readthedocs.io/en/latest/register/parameters.html) or [resolve](https://autofac.readthedocs.io/en/latest/resolve/parameters.html) will be considered when injecting required properties. However, `NamedParameter` and `PositionalParameter` are not considered valid parameters for property injection, since they are considered to only apply to constructor parameters.
在註冊或解析時提供的任何 `TypedParameter` 都會在注入必要屬性時被考慮。然而， `NamedParameter` 和 `PositionalParameter` 被認為不是有效的屬性注入參數，因為它們被認為僅適用於構造函數參數。

### PropertiesAutowired

You can use the `PropertiesAutowired()` modifier at registration time to inject properties on any component:
您可以在註冊時使用 `PropertiesAutowired()` 修飾詞來注入任何元件的屬性：

```csharp
// Default behavior: inject all properties that are public and writable.
// 預設行為：注入所有公共且可寫入的屬性。
builder.RegisterType<A>().PropertiesAutowired();

// Provide a delegate property selector to be more granular. This example
// shows injecting all properties where the property type starts with
// 'I' - one way you might "only inject interface properties." The delegate
// gets the PropertyInfo describing the property to be injected and the
// instance getting injected.
// 提供一個委託屬性選擇器，以便更細緻地進行選擇。
// 此範例展示如何注入所有屬性類型以「I」開頭的屬性——這是一種「僅注入介面屬性」的方法。
// 委託會取得描述要注入的屬性和實例的 PropertyInfo。
builder.RegisterType<B>()
       .PropertiesAutowired(
            (propInfo, instance) => propInfo.PropertyType.Name.StartsWith("I"));

// Even more fancy, you can provide your own implementation of
// IPropertySelector with as much functionality as you want. Don't
// forget this will run on every associated resolution, so performance
// is important!
// 更高級的做法是，您可以提供自訂的 IPropertySelector 實現，並根據需要添加任何功能。
// 請記住，此操作將在每個關聯的解析度上運行，因此效能至關重要！
builder.RegisterType<C>().PropertiesAutowired(new MyCustomPropSelector());
```

### Manually Specifying Properties 手動指定屬性

If you have one specific property and value to wire up, you can use the `WithProperty()` modifier:
如果您有一個特定的屬性和值要連接，您可以使用 `WithProperty()` 修飾詞：

```csharp
builder.RegisterType<A>().WithProperty("PropertyName", propertyValue);
```

### Overriding Required Properties 覆寫必要的屬性

Any property values provided for required properties using the `WithProperty` method when registering a type will override the requirement to inject that property, and Autofac will use the provided value instead:
使用 `WithProperty` 方法在註冊類別時，為必須屬性提供的任何屬性值都會覆蓋注入該屬性的需求，Autofac 會使用提供的值：

```csharp
public class MyComponent
{
    public required ILogger Logger { protected get; init; }
    
    public required IConfigReader ConfigReader { protected get; init; }
}

var builder = new ContainerBuilder();
builder.RegisterType<MyComponent>().WithProperty("Logger", new ConsoleLogger());

var container = builder.Build();

// This will not throw, despite ILogger not being registered.
// The Logger property is provided by WithProperty.
// 即使 ILogger 未註冊，此操作也不會拋出例外。
// Logger 屬性由 WithProperty 提供。
container.Resolve<MyComponent>();
```

### Injecting Properties on an Existing Object 在現有物件上注入屬性

You can also populate _just the properties_ on an object. Do this using the `InjectUnsetProperties` extension on a lifetime scope, which will resolve and populate properties that are _public, writable, and not yet set (null)_:
您也可以僅僅填充物件上的屬性。使用 `InjectUnsetProperties` 擴充功能在生命週期範圍上執行此操作，它將解析並填充公共、可寫入且尚未設置（null）的屬性：

```csharp
lifetimeScope.InjectUnsetProperties(myObject);
```

## Method Injection 方法注入

The simplest way to call a method to set a value on a component is to use a [lambda expression component](https://autofac.readthedocs.io/en/latest/register/registration.html#register-registration-lambda-expression-components) and handle the method call right in the activator:
呼叫方法來在元件上設定值的最簡單方式是使用 lambda 表達式元件，並在活躍器中直接處理方法呼叫：

```csharp
builder.Register(c => {
    var result = new MyObjectType();
    var dep = c.Resolve<TheDependency>();
    result.SetTheDependency(dep);
    return result;
});
```

If you can’t use a registration lambda, you can add an [activating event handler](https://autofac.readthedocs.io/en/latest/lifetime/events.html):
如果你無法使用註冊 lambda，你可以添加一個活躍事件處理程序：

```csharp
builder
    .RegisterType<MyObjectType>()
    .OnActivating(e => {
        var dep = e.Context.Resolve<TheDependency>();
        e.Instance.SetTheDependency(dep);
    });
```

# Assembly Scanning 組裝掃描

Autofac can use conventions to find and register components in assemblies. You can scan and register individual types or you can scan specifically for [Autofac modules](https://autofac.readthedocs.io/en/latest/configuration/modules.html).
Autofac 可以使用規則來尋找並註冊組裝中的組件。您可以掃描並註冊個別型別，或者您可以特別掃描 Autofac 模組。

## Scanning for Types 掃描型別

Otherwise known as convention-driven registration or scanning, Autofac can register a set of types from an assembly according to user-specified rules:
也稱為規則驅動註冊或掃描，Autofac 可以根據使用者指定的規則，從組裝中註冊一組型別：

```csharp
var dataAccess = Assembly.GetExecutingAssembly();

builder.RegisterAssemblyTypes(dataAccess)
       .Where(t => t.Name.EndsWith("Repository"))
       .AsImplementedInterfaces();
```

Each `RegisterAssemblyTypes()` call will apply one set of rules only - multiple invocations of `RegisterAssemblyTypes()` are necessary if there are multiple different sets of components to register.
每個 `RegisterAssemblyTypes()` 呼叫只會套用一套規則 - 如果要註冊多個不同組的組件，則需要多次呼叫 `RegisterAssemblyTypes()` 。

Autofac also supports assembly scanning for open generics using `RegisterAssemblyOpenGenericTypes()`. This uses the same general semantics as `RegisterAssemblyTypes()`:
Autofac 也支援使用 `RegisterAssemblyOpenGenericTypes()` 進行開放泛型的組件掃描。這使用與 `RegisterAssemblyTypes()` 相同的通用語義：

```csharp
var dataAccess = Assembly.GetExecutingAssembly();

builder.RegisterAssemblyOpenGenericTypes(dataAccess)
       .Where(t => t.Name.StartsWith("MessageHandler"))
       .AsImplementedInterfaces();
```

### Filtering Types 過濾類型

`RegisterAssemblyTypes()` and `RegisterAssemblyOpenGenericTypes()` each accept a parameter array of one or more assemblies. By default, **all concrete classes in the assembly will be registered.** This includes internal and nested private classes. You can filter the set of types to register using some provided LINQ-style predicates.
`RegisterAssemblyTypes()` 和 `RegisterAssemblyOpenGenericTypes()` 各自接受一個包含一個或多個組件的參數陣列。預設情況下，組件中的所有具體類別都會被註冊。這包括內部類別和嵌套的私有類別。您可以使用提供的 LINQ 風格 predicat 來過濾要註冊的類型集合。

In 4.8.0 a `PublicOnly()` extension was added to make data encapsulation easier. If you only want your public classes registered, use `PublicOnly()`:
在 4.8.0 中新增了 `PublicOnly()` 擴充功能，讓數據封裝更方便。如果您只想註冊公共類別，請使用 `PublicOnly()` ：

```csharp
builder.RegisterAssemblyTypes(asm)
       .PublicOnly();
```

To apply custom filtering to the types that are registered, use the `Where()` predicate:
若要對已註冊的類型套用自訂篩選，請使用 `Where()` 語句：

```csharp
builder.RegisterAssemblyTypes(asm)
       .Where(t => t.Name.EndsWith("Repository"));
```

To exclude types from scanning, use the `Except()` predicate:
若要排除某些類型不被掃描，請使用 `Except()` 語句：

```csharp
builder.RegisterAssemblyTypes(asm)
       .Except<MyUnwantedType>();
```

The `Except()` predicate also allows you to customize the registration for the specific excluded type:
`Except()` 語句也允許您針對特定排除的類型自訂註冊：

```csharp
builder.RegisterAssemblyTypes(asm)
       .Except<MyCustomizedType>(ct =>
          ct.As<ISpecial>().SingleInstance());
```

Multiple filters can be used, in which case they will be applied with logical AND.
多個濾鏡可以同時使用，此時它們將以邏輯 AND 的方式應用。

```csharp
builder.RegisterAssemblyTypes(asm)
       .PublicOnly()
       .Where(t => t.Name.EndsWith("Repository"))
       .Except<MyUnwantedRepository>();
```

### Specifying Services 指定服務

The registration syntaxes for `RegisterAssemblyTypes()` and `RegisterAssemblyOpenGenericTypes()` are a subset of [the registration syntax for single types](https://autofac.readthedocs.io/en/latest/register/index.html), so methods like `As()` all work with assemblies as well:
`RegisterAssemblyTypes()` 和 `RegisterAssemblyOpenGenericTypes()` 的註冊語法是單一類型註冊語法的子集，因此像 `As()` 這樣的方法也適用於組件：

```csharp
builder.RegisterAssemblyTypes(asm)
       .Where(t => t.Name.EndsWith("Repository"))
       .As<IRepository>();
```

Additional overloads to `As()` and `Named()` accept lambda expressions that determine, for a type, which services it will provide:
`As()` 和 `Named()` 的額外重載接受 lambda 表達式，該表達式為每個類型決定它將提供哪些服務：

```csharp
builder.RegisterAssemblyTypes(asm)
       .As(t => t.GetInterfaces()[0]);
```

As with normal component registrations, multiple calls to `As()` are added together.
與一般元件註冊相同，多個 `As()` 的呼叫會疊加起來。

A number of additional registration methods have been added to make it easier to build up common conventions:
為了讓常見的規範更容易建立，已新增多種額外的註冊方法：

| Method  方法                  | Description  描述                                                                                                                                                    | Example                                                                                                                                                   |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `AsImplementedInterfaces()` | Register the type as providing all of its public interfaces as services (excluding `IDisposable`).  <br>註冊類型，將其所有公共介面作為服務提供（排除 `IDisposable` ）。                    | %% Tite %%<br>```<br>builder.RegisterAssemblyTypes(asm)<br>       .Where(t => t.Name.EndsWith("Repository"))<br>       .AsImplementedInterfaces();<br>``` |
| `AsClosedTypesOf(open)`     | Register types that are assignable to a closed instance of the open generic type.  <br>註冊可指派給開放泛型類型封閉實例的類型。                                                        | %% Tite %%<code>builder.RegisterAssemblyTypes(asm)<br>       .AsClosedTypesOf(typeof(IRepository<>));<br></code>                                          |
| `AsSelf()`                  | The default: register types as themselves - useful when also overriding the default with another service specification.  <br>預設值：將類型註冊為其本身 - 當同時覆寫預設值為另一個服務規範時很有用。 | <code>builder.RegisterAssemblyTypes(asm)<br>       .AsImplementedInterfaces()<br>       .AsSelf();<br></code>                                             |

## Scanning for Modules 掃描模組

Module scanning is performed with the `RegisterAssemblyModules()` registration method, which does exactly what its name suggests. It scans through the provided assemblies for [Autofac modules](https://autofac.readthedocs.io/en/latest/configuration/modules.html), creates instances of the modules, and then registers them with the current container builder.
模組掃描是透過 `RegisterAssemblyModules()` 註冊方法來執行的，它的名稱就說明了它的功能。它會掃描所提供的組合中是否有 Autofac 模組，創建模組的實例，然後將它們註冊到當前的容器建構器中。

For example, say the two simple module classes below live in the same assembly and each register a single component:
例如，假設下面兩個簡單的模組類別存在於同一個組合中，且每個模組註冊一個元件：

```csharp
public class AModule : Module
{
    protected override void Load(ContainerBuilder builder)
    {
        builder.Register(c => new AComponent()).As<AComponent>();
    }
}

public class BModule : Module
{
    protected override void Load(ContainerBuilder builder)
    {
        builder.Register(c => new BComponent()).As<BComponent>();
    }
}
```

The overload of `RegisterAssemblyModules()` that _does not accept a type parameter_ will register all classes implementing `IModule` found in the provided list of assemblies. In the example below **both modules** get registered:
不接受型別參數的 `RegisterAssemblyModules()` 註冊方法會註冊在所提供的組合清單中實作了 `IModule` 的所有類別。在下面的範例中，兩個模組都會被註冊：

```csharp
var assembly = typeof(AComponent).Assembly;
var builder = new ContainerBuilder();

// Registers both modules
builder.RegisterAssemblyModules(assembly);
```

The overload of `RegisterAssemblyModules()` with _the generic type parameter_ allows you to specify a base type that the modules must derive from. In the example below **only one module** is registered because the scanning is restricted:
具有泛型型別參數的 `RegisterAssemblyModules()` 註冊方法允許您指定模組必須繼承的基底型別。在下面的範例中，僅有一個模組被註冊，因為掃描受到了限制：

```csharp
var assembly = typeof(AComponent).Assembly;
var builder = new ContainerBuilder();

// Registers AModule but not BModule
builder.RegisterAssemblyModules<AModule>(assembly);
```

The overload of `RegisterAssemblyModules()` with _a Type object parameter_ works like the generic type parameter overload but allows you to specify a type that might be determined at runtime. In the example below **only one module** is registered because the scanning is restricted:
`RegisterAssemblyModules()` 的型別物件參數重載功能與泛型型別參數重載類似，但允許您指定在執行時可能才會確定的型別。在以下範例中僅註冊了一個模組，因為掃描受到限制：

```csharp
var assembly = typeof(AComponent).Assembly;
var builder = new ContainerBuilder();

// Registers AModule but not BModule
builder.RegisterAssemblyModules(typeof(AModule), assembly);
```

## IIS Hosted Web Applications IIS 主機網站應用程式

When using assembly scanning with IIS applications, you can run into a little trouble depending on how you do the assembly location. ([This is one of our FAQs](https://autofac.readthedocs.io/en/latest/faq/iis-restart.html))
在使用組合掃描與 IIS 應用程式時，您可能會遇到一些麻煩，取決於您如何設定組合位置。（這是我們的常見問題之一）

When hosting applications in IIS all assemblies are loaded into the `AppDomain` when the application first starts, but **when the AppDomain is recycled by IIS the assemblies are then only loaded on demand.**
當在 IIS 主機應用程式時，所有組合在應用程式首次啟動時都會載入到 `AppDomain` 中，但當 AppDomain 被 IIS 回收時，組合則是按需載入。

To avoid this issue use the [GetReferencedAssemblies()](https://msdn.microsoft.com/en-us/library/system.web.compilation.buildmanager.getreferencedassemblies.aspx) method on [System.Web.Compilation.BuildManager](https://msdn.microsoft.com/en-us/library/system.web.compilation.buildmanager.aspx) to get a list of the referenced assemblies instead:
要避免這個問題，請使用 System.Web.Compilation.BuildManager 的 GetReferencedAssemblies() 方法來取得參考組件的清單，而不是：

```csharp
var assemblies = BuildManager.GetReferencedAssemblies().Cast<Assembly>();
```

That will force the referenced assemblies to be loaded into the `AppDomain` immediately making them available for module scanning.
這將強制參考組件被載入到 `AppDomain` 中，並立即讓它們可供模組掃描使用。
