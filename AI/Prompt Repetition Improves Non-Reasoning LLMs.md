# 《複製貼上就能暴增準確率：Google 新論文揭示的因果注意力結構性缺陷》

Google Research 在 2025 年 12 月發表的一篇論文《Prompt Repetition Improves Non-Reasoning LLMs》，在技術圈引發了相當有趣的討論。這篇論文的核心發現簡單到令人懷疑：只要把你的 Prompt（提示詞）完整複製貼上一次，形成「\<QUERY>\<QUERY>」的結構，就能在不使用思維鏈（Chain-of-Thought）的情況下，顯著提升模型在特定任務上的表現。

這聽起來像是一個粗糙的 Prompt Hack，甚至有點像都市傳說。然而，經過 Gemini、GPT-4o、Claude 3 與 DeepSeek 等主流模型的廣泛測試，研究團隊給出了具有統計顯著性的數據支持。在他們測試的 70 組模型與任務組合中，這種簡單的重複策略取得了 47 次顯著勝利，且沒有出現任何性能倒退的案例。

作為一名長期關注 LLM 架構的工程師，我認為這份報告的價值不在於它提供了一個「複製貼上」的技巧，而在於它無意間暴露了當前 Transformer 架構（特別是 Decoder-only 模型）在處理注意力機制時的底層缺陷。這不是魔法，這是對「因果遮罩」（Causal Masking）限制的一種暴力破解。

# 過去我們對 Prompt Engineering 的慣性假設

在理解為什麼「重複」有效之前，我們得先回顧一下產業界在 Prompt 優化上的幾個既定習慣，這些習慣大多建立在對模型運作方式的直觀（但也許不完全正確）理解上：

## 「精簡至上」的成本考量

我們習慣盡可能縮短 Prompt，移除不必要的冗餘資訊。這背後的邏輯很簡單：輸入 token 越少，API 費用越低，且我們假設模型能像人類一樣，只要看過一次就能記住關鍵字。

## 假設模型具備「全局視野」

我們傾向認為，只要資訊在 Context Window（上下文視窗）內，模型就能平等地「看見」所有內容。因此，我們花大量時間在調整 RAG（檢索增強生成）的排序，試圖把最相關的文檔放在最前面或最後面，卻忽略了模型在閱讀中間段落時的注意力衰減。

## 依賴 CoT 解決所有理解問題

當模型答錯時，我們的標準反應是加上 "Let's think step by step"。這確實有效，但也帶來了副作用：大幅增加的輸出 token 數量，導致推論延遲（Latency）變長，且成本成倍增加。

# 新發現帶來的結構性衝擊與技術解構

這篇論文的發現，實際上是對上述慣性假設的一次修正。它揭示了 LLM 在「閱讀」與「理解」之間的斷裂。以下是四個關鍵的技術洞察：

## 因果模型的單向視線如同無法回頭的輸送帶

目前的 LLM 絕大多數是因果語言模型（Causal LLMs）。這意味著模型在處理第 N 個 token 時，只能看見第 1 到 N-1 個 token，完全無法預知第 N+1 個 token 的內容。

你可以把這想像成一條單向通行的工廠輸送帶。當模型讀到 Prompt 前半段的關鍵資訊時，它還不知道這條資訊對後面的問題有多重要，因此分配的注意力權重可能很低。等到讀完後面的問題想回頭找答案時，前面的資訊特徵可能已經在層層傳遞中被稀釋了。

## 重複輸入是建立人工雙向注意力的後照鏡

透過將 Prompt 重複一次（<QUERY 1><QUERY 2>），我們實際上是在模擬 BERT 等 Encoder 模型的「雙向注意力」。

當模型處理第二遍內容（QUERY 2）時，每一個 token 都能透過注意力機制「看見」第一遍內容（QUERY 1）的所有細節。這就像是在輸送帶的末端裝了一面巨大的後照鏡，讓模型在處理第二遍資訊時，擁有對第一遍資訊的完整全局視野。這解釋了為什麼在 NameIndex 這種極端依賴精準檢索的測試中，Gemini 2.0 Flash-Lite 的準確率能從慘不忍睹的 21.33% 直接跳升到 97.33%。

## 預填充與解碼的成本不對稱性

許多人擔心的問題是：「輸入加倍，難道不會變慢嗎？」這觸及了 LLM 推論的硬體特性。

LLM 的推論分為兩個階段：預填充（Prefill）與解碼（Decode）。預填充階段是高度平行化的，GPU 可以像同時讓幾千名乘客上一台列車那樣，極快地處理輸入 token。而解碼階段則是序列化的，必須一個字一個字生成，就像乘客必須排隊單一出口下車。

論文數據顯示，增加輸入長度對總延遲的影響微乎其微（毫秒級差異），因為瓶頸通常卡在解碼階段。這就像是把水管的進水口加寬（預填充），並不會影響出水口原本就固定的流速（解碼），但水流的品質卻變好了。

## 推理模型已經在偷偷重複

論文指出，針對經過強化學習（RL）訓練的推理模型（如 o1 或開啟 thinking 模式的模型），外部手動重複 Prompt 的效果不明顯。

我合理推測，這是因為這些推理模型在內部的思維鏈（Chain-of-Thought）過程中，往往會自發性地複述使用者的問題或限制條件。換句話說，System 2 的「慢思考」本質上就是一種「內在的重複與反芻」。這篇論文提出的方法，其實就是用低成本的輸入 token，去模擬昂貴的輸出端推理行為。

# 產業影響評估：誰受惠，誰受傷？

這項技術的落地門檻極低，預計會對幾個領域產生立即影響：

## 對於 RAG 系統開發者

這是一個巨大的利多。過去為了解決 "Lost in the Middle"（迷失在中間）現象，開發者需要設計複雜的重排序演算法。現在，對於關鍵的檢索文檔或複雜指令，簡單的重複輸入可能是一個更具性價比的解決方案。特別是在處理大量非結構化資料的提取任務時，這種方法的 ROI 極高。

## 對於 Agent 與自動化工作流

Agent 最怕的就是不聽指令或格式錯誤。重複 Prompt 實際上增強了指令在模型內部的權重。對於那些需要嚴格遵循 JSON 格式輸出的任務，這種方法可能會顯著降低解析錯誤（Parsing Error）的機率，讓 Agent 的穩定性提升。

## 對於 API 成本結構

這對使用者來說是精打細算的勝利。目前的 API 定價通常是「輸入便宜，輸出貴」（比例約為 1:3 甚至 1:4）。使用 Prompt 重複策略，我們消耗的是廉價的輸入 token，節省的是昂貴且緩慢的 CoT 輸出 token。這是在不升級模型規格的前提下，榨出模型潛能的經濟手段。

# 結語：通往下一代架構的過渡

雖然「複製貼上」看起來很土法煉鋼，但它實際上是在修補 Transformer 架構中「因果遮罩」帶來的先天缺陷。這也讓我思考，未來的模型架構是否會直接在底層整合這種「回看」機制？例如，未來的 LLM 是否會內建類似 Recurrent Transformer 的循環處理單元，讓模型在生成第一個字之前，自動在內部對輸入進行多輪的 Attention 掃描，而不需要使用者手動貼上兩次？

在那個時代來臨之前，作為工程師的我們，或許應該重新審視手上的 Prompt。有時候，最有效的優化不是更華麗的詞藻，而僅僅是再說一遍。

思考題：如果輸入端的重複能達到類似推理模型的效果，這是否意味著目前的 CoT 技術有很大一部分算力，其實只是在補償模型「沒看清楚題目」的缺陷，而非真正的邏輯推演？
