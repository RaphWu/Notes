- 以此圖為例，在製作 class diagram 時：
	- 表頭將會是此類別名稱 ( 圖 1 )
	- 其中如果表頭以 Italics 字體樣式表示也就是斜體，就代表為抽象類別。( 圖 2 )
	- 而如果是介面，將會標示出 `<<interface>>` 字樣。 ( 圖 3 )
- 而其下成員會分為兩區塊標明
    - 上方區塊會標明所屬欄位（ Fields )
    - 下方區塊會標明所屬方法 ( Methods )
    - 其中又以正負號作為欄位的存取權限差異

![[UML Diagram 軟工溝通好幫手_04.webp|圖 4]]

而此圖當中的虛實線搭配箭頭的型態分別代表著不一樣的關聯性 (圖片中由上到下由左至右由弱到強的關聯性)

- **Dependency 依賴**以 實心箭頭虛線表示
    - 物件 A 依賴於物件 B ( 被使用的一方為箭頭所在位置 ），
    - 代表者物件 Ａ 會受到物件 Ｂ 的改動所影響。
    - B 在 Ａ 中被使用屬於**局部性**的，如只能於某一方法中存取。
- **Association 關聯**以 實心箭頭實線表示
    - 物件 Ａ 關聯於物件 B ，屬於比依賴更強一點的關聯性，
    - 代表者物件 Ａ 會受到物件 Ｂ 的改動所影響，並且物件 A ==總是能知道==物件 B 開放出來的介面。
    - B 在 Ａ 中被使用屬於**全域性**的。
- **Aggregation 聚合**以 空心菱形實線表示 （ has-a 關係 ）
    - 物件 Ａ 聚合於物件 B ，代表者物件 Ａ **擁有著**物件 Ｂ，並且**由他組成**，
    - 但物件 Ｂ **可以單獨存在**者，並且物件 Ｂ 的生命週期並**不受**物件 Ａ 所控制。
    - 與關聯 Association 差異在於 ==物件 Ａ 是否需要有 物件 B 才有存在價值==。
- **composition 組合**以 實心菱形實線表示（ contains-a 關係 ）
    - 物件 Ａ 組合於物件 B，代表者物件 Ａ **擁有著**物件 Ｂ，並且**由他組成**，
    - 但物件 Ｂ **不可單獨存在**，單獨存在沒有意義，且物件 B 的生命週期**受**物件 Ａ 所影響。
- **implementation 實作**以 空心箭頭虛線表示
    - 物件 Ａ 定義宣告在介面 B 中的所有成員，且物件 Ａ **可以被當作**介面 B 使用。
- **inheritance 繼承**以 空心箭頭實線表示 （ is-a 關係 ）
    - 物件 A 繼承於物件 B ，且物件 Ａ **可以被當作**物件 B 使用，但物件 Ａ 能夠基於物件Ｂ的實作跟介面**加以擴展**。
