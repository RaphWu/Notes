# 已經有生成程式碼了，如何提出新的 Prompt 執行重構？

可以直接貼 Prompt 要求 Copilot 重構，但**結果取決於你怎麼下指令**。
預設情況下，Copilot 比較傾向「增量修改」，不會主動整個砍掉重建，除非你明確要求。

我分成三種情境說明。

## 一、如果你只貼新的 Prompt

常見行為：

- 在現有架構上補功能
- 插入新的方法
- 修補缺少的 try-catch
- 加幾個欄位與背景 Task
- 不會重構整體設計

風險：

- 舊架構問題會被保留
- 可能出現雙重 Read Loop
- 狀態機變成「疊加式修補」
- 競態控制可能混亂

結論：
這是「補丁式強化」，不是重構。

## 二、如果你要「真正重構」，必須明確指示

你需要明確寫：

- 允許重構現有類別
- 若架構不符合要求可重新設計
- 可以調整方法簽名
- 可以刪除舊實作
- 以新的 Prompt 規格為唯一標準

例如這樣的語氣：

```markdown
請依下列完整規格重構現有 SerialPortService。
若現有設計不符合規格，允許重新設計類別結構。
可刪除或調整不符合規範的程式碼。
以本規格為最終標準。
```

```markdown
允許進行破壞性重構，不保留舊版 API 相容性。
允許修改 csproj、新增套件、重組檔案結構、重新設計 public API、建立 Native 封裝層與 Fake/Simulator、加入 XML 文件規範與 CONTRIBUTING 文件。
```

這樣 Copilot 才會真正「架構級重構」。

## 三、是否會整個砍掉重建？

取決於你：

- 如果只說「請依規格調整」→ 增量修改
- 如果說「請重構，允許重新設計」→ 會大幅改動
- 如果說「請從零重新生成符合規格版本」→ 幾乎重寫

## 建議策略（專業做法）

不要一次大爆改。

建議流程：

1. 先請 Copilot 分析現有程式碼與規格差異
2. 列出不符合項目
3. 再逐項重構
4. 最後才全面重整

這樣比較穩定，不容易破壞可用部分。

## 如果你現在問我專業建議

如果你的現有程式碼：

- 沒有持續 Read Loop
- 沒有 Write 失敗即 Fault
- 狀態機鬆散
- 背景 Task 可能 crash

那我會建議：

👉 直接允許架構級重構
比修修補補安全。

如果只是缺少：

- 裝置消失檢查
- 錯誤類型細分
- 多幾個 try-catch

那可以增量改造。

## 關鍵觀念

Copilot 不會主動推翻你原本的架構
除非你授權它這麼做。
