# 在 ASP.NET Core 中實作 RBAC + ABAC 混合授權邏輯

## 一、設計思路總覽

RBAC 管角色，ABAC 判條件。
整體流程如下：

```scss
User 登入 → 產生 ClaimsPrincipal
→ 授權檢查 (AuthorizationHandler)
→ 同時驗證 Role 與屬性條件 (Policies)
→ 決定是否允許執行動作
```

## 二、範例資料模型設計（概念）

```csharp
public class User
{
    public string UserName { get; set; }
    public string Department { get; set; }
    public string Location { get; set; }
    public List<string> Roles { get; set; } = new();
}
```

範例：

```csharp
Alice：Roles=[Manager], Department="HR", Location="Office"
```

## 三、RBAC 授權（角色基礎）

在 Controller 層使用 `[Authorize(Roles="Manager")]` 進行角色檢查。

```csharp
[Authorize(Roles = "Manager")]
[HttpPost("approve-leave")]
public IActionResult ApproveLeave() => Ok("Approved");
```

這部分由 ASP.NET Core 的角色授權機制自動處理，屬於 RBAC 部分。

## 四、ABAC 授權（屬性判斷）

要讓系統能進一步檢查條件（如部門與地點），需建立 **自訂 Policy 與 Handler**。

### 1. 定義 Policy Requirement

```csharp
public class DepartmentRequirement : IAuthorizationRequirement
{
    public string Department { get; }
    public DepartmentRequirement(string department) => Department = department;
}
```

### 2. 撰寫授權邏輯（Handler）

```csharp
public class DepartmentHandler : AuthorizationHandler<DepartmentRequirement>
{
    protected override Task HandleRequirementAsync(
        AuthorizationHandlerContext context,
        DepartmentRequirement requirement)
    {
        var userDept = context.User.FindFirst("Department")?.Value;

        if (userDept == requirement.Department)
            context.Succeed(requirement);

        return Task.CompletedTask;
    }
}
```

### 3. 註冊 Policy

```csharp
builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("HRDepartmentOnly", policy =>
        policy.RequireRole("Manager")
              .AddRequirements(new DepartmentRequirement("HR")));
});

builder.Services.AddSingleton<IAuthorizationHandler, DepartmentHandler>();
```

### 4. 控制器使用 Policy

```csharp
[Authorize(Policy = "HRDepartmentOnly")]
[HttpPost("approve-leave")]
public IActionResult ApproveLeave() => Ok("HR Manager Approved");
```

## 五、Claims 與登入建立

登入成功後，在建立 ClaimsPrincipal 時加入必要屬性：

```csharp
var claims = new List<Claim>
{
    new Claim(ClaimTypes.Name, "Alice"),
    new Claim(ClaimTypes.Role, "Manager"),
    new Claim("Department", "HR"),
    new Claim("Location", "Office")
};

var identity = new ClaimsIdentity(claims, "CustomAuth");
var principal = new ClaimsPrincipal(identity);
await HttpContext.SignInAsync(principal);
```

## 六、混合邏輯運作過程

1. 使用者登入 → 系統建立角色與屬性 Claims
2. 呼叫 API → `[Authorize]` 驗證角色（RBAC）
3. 通過角色驗證後 → 自訂 `Handler` 檢查屬性（ABAC）
4. 若符合條件 → `context.Succeed()` 通過授權

簡化流程圖：

```csharp
User → [Role Check] → [Attribute Check] → Access Granted
```

## 七、延伸設計方向

1. **多屬性驗證**：同時檢查 Department、Location、Time。
2. **外部化 Policy**：用 JSON 或資料庫儲存條件規則。
3. **使用 Casbin 或 OPA**：統一管理多系統授權政策。
4. **增加稽核（Audit）**：記錄授權通過或拒絕的原因。

# 外部化 Policy：以 JSON 或資料庫動態管理 RBAC + ABAC 授權規則

## 一、設計目標

讓授權邏輯不依賴硬編碼（hardcoded），而是從外部設定檔或資料庫載入。
這樣管理員可在不重新部署程式的情況下修改規則。

## 二、核心概念

外部化的授權規則通常具備以下要素：

1. **Role（角色）**：RBAC 基礎授權條件
2. **Attributes（屬性）**：ABAC 條件，如部門、地點、時間
3. **Actions（行為）**：允許執行的操作名稱或 API
4. **Policies（政策）**：綜合判斷條件與結果（Allow / Deny）

## 三、JSON 格式範例

```json
[
    {
        "PolicyName": "HRManagerPolicy",
        "Roles": ["Manager"],
        "Attributes": {
            "Department": "HR",
            "Location": "Office"
        },
        "Actions": ["ApproveLeave", "ViewEmployeeRecord"],
        "Effect": "Allow"
    },
    {
        "PolicyName": "RemoteRestriction",
        "Roles": ["Manager"],
        "Attributes": {
            "Location": "Remote"
        },
        "Actions": ["ViewEmployeeRecord"],
        "Effect": "Deny"
    }
]
```

說明：

- 第一條允許 HR 部門的 Manager 在公司內部操作兩項功能。
- 第二條禁止任何遠端登入的 Manager 檢視員工資料。

## 四、讀取與解析 JSON

```csharp
public class PolicyRule
{
    public string PolicyName { get; set; }
    public List<string> Roles { get; set; } = new();
    public Dictionary<string, string> Attributes { get; set; } = new();
    public List<string> Actions { get; set; } = new();
    public string Effect { get; set; } // "Allow" or "Deny"
}
```

載入設定：

```csharp
var policyJson = File.ReadAllText("policies.json");
var policies = JsonSerializer.Deserialize<List<PolicyRule>>(policyJson);
```

## 五、動態授權檢查流程

```csharp
public class DynamicPolicyService
{
    private readonly List<PolicyRule> _policies;

    public DynamicPolicyService(List<PolicyRule> policies)
    {
        _policies = policies;
    }

    public bool IsAuthorized(ClaimsPrincipal user, string action)
    {
        var userRole = user.FindFirst(ClaimTypes.Role)?.Value;
        var userDept = user.FindFirst("Department")?.Value;
        var userLoc  = user.FindFirst("Location")?.Value;

        foreach (var policy in _policies)
        {
            if (policy.Roles.Contains(userRole) &&
                policy.Actions.Contains(action))
            {
                bool match = true;
                foreach (var attr in policy.Attributes)
                {
                    var value = attr.Key switch
                    {
                        "Department" => userDept,
                        "Location" => userLoc,
                        _ => null
                    };
                    if (value != attr.Value) match = false;
                }

                if (match)
                    return policy.Effect.Equals("Allow", StringComparison.OrdinalIgnoreCase);
            }
        }
        return false;
    }
}
```

## 六、使用範例

```csharp
var authorized = dynamicPolicyService.IsAuthorized(User, "ApproveLeave");

if (authorized)
    return Ok("Access Granted");
else
    return Forbid();
```

## 七、資料庫化策略（進階）

1. 將 JSON 結構轉為資料表：
    - **Policies**
    - **PolicyRoles**
    - **PolicyAttributes**
    - **PolicyActions**

2. 從資料庫動態載入後快取（MemoryCache 或 Redis）
3. 可於後台提供「即時更新策略」的管理介面

## 八、思路延伸

這樣的設計可逐步演化成專業授權系統，例如：

- **OPA（Open Policy Agent）**：以 Rego 語言撰寫政策
- **Casbin**：支援 RBAC、ABAC、PBAC 等模式
- **自訂 JSON Policy Engine**：整合 ASP.NET Core 的 Middleware 授權流程
