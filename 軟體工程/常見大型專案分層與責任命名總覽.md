# 常見大型專案分層與責任命名總覽

以下整理在企業級或大型系統中，常見的專案層級（Layer）或模組（Module）命名方式，以及各自**典型責任範圍**與**是否常被做成獨立 Library**。

## Core（或 Domain）

- 業務核心模型與規則
- Entity、Value Object、Domain Service
- 不依賴任何 UI、資料庫、Framework
- 專案中最穩定、最不易變動
- 幾乎一定是獨立 Library

## Application

- 使用案例（Use Case）與流程協調
- Application Service、Command / Query
- 定義對外操作行為，不含技術細節
- 依賴 Core，但不依賴 Infrastructure
- 通常為獨立 Library

## Infrastructure

- 技術實作層
- Repository 實作、ORM、檔案、網路、硬體、外部 API
- Logging、Cache、Message Queue、Serial Port
- 依賴 Application / Core
- 幾乎一定獨立成 Library，且可替換

## Presentation / UI

- 使用者介面與互動邏輯
- WinForm、WPF、Web、API Controller
- View、ViewModel、Presenter
- 只負責顯示與輸入，不含商業邏輯
- 通常是 App（EXE / Web Host）

## Framework

- 橫向技術框架與平台級封裝
- DI、Logging、Configuration、Task Scheduling
- 對特定技術棧的標準化封裝
- 多個系統共用
- 幾乎一定是外部 Library 或 NuGet

## Common / Shared

- 通用工具與輔助類型
- Extension Methods、Helper、Primitive Wrappers
- 無明確業務語意
- 依賴最少，容易被濫用
- 建議嚴格控管內容

## Cross-Cutting / Abstractions

- 橫切關注點的抽象定義
- Logging 介面、Clock、Id Generator
- 通常只有 Interface 與契約
- 常被拆成 *.Abstractions 專案

## Contracts / DTO / Models

- 系統間資料交換模型
- API Request / Response、Event Payload
- 不含邏輯，僅資料結構
- 常獨立成 Library 供雙方引用

## Services（或 Integrations）

- 對外系統整合
- ERP、MES、Payment、Third-party SDK 封裝
- 可視為 Infrastructure 的子分類
- 規模大時獨立命名較清楚

## Bootstrap / Host / Startup

- 應用程式啟動與組裝
- DI 註冊、模組載入、設定綁定
- Main / Program / Startup 類型
- 通常只存在於 App 專案

## Plugins / Modules

- 可插拔式功能模組
- 動態載入、版本可替換
- 常用於設備、演算法、客製流程
- 多為獨立 Library

## Tests

- 單元、整合、自動化測試
- Unit / Integration / E2E
- 不部署到正式環境
- 專屬測試專案

## 命名與結構常見對照

- Core ≈ Domain
- Application ≈ UseCases
- Infrastructure ≈ Adapters / Implementations
- Presentation ≈ UI / API / Client
- Framework ≈ Platform / Foundation
- Common ≈ Utilities（需高度節制）

## 實務命名建議

- 名稱必須反映「責任」，不是技術名詞
- 穩定度越高，越靠近 Core
- 變動越頻繁，越靠近外層
- Common 若開始出現業務語意，代表分層已失敗
- 能 Interface 的地方，優先放在內層

## 常見專案結構範例

- Xxx.Domain
- Xxx.Application
- Xxx.Application.Abstractions
- Xxx.Infrastructure
- Xxx.Infrastructure.Sql
- Xxx.Presentation.WinForm
- Xxx.Framework.Logging
- Xxx.Framework.DI
- Xxx.Contracts
- Xxx.Tests.Unit

如果你有特定情境，例如「WinForm + 多設備控制」、「NuGet 化共用框架」、「單體但模組化」或「需要避免 Circular Reference」，我可以依你的實際架構幫你畫出建議分層與命名策略。

# 在企業級 .NET 專案中，不同層級命名在宣告什麼責任與穩定性

以下直接給你**業界實務可落地的命名慣例對照表**，以及何時該用、何時不該用。

## 一、最常見的層級命名與實際語意

### Core

#### 慣例語意

- 業務核心
- 領域模型
- 穩定、長壽、變動成本高

#### 通常包含

- Domain Model
- Entity / ValueObject
- Domain Service
- 業務規則
- 與 UI、DB、Framework 無關的邏輯

#### 不應包含

- Logging
- DI
- EF / Dapper
- IO、File、Network

#### 命名範例

- `Company.Product.Core`
- `Company.Product.Domain`
- `Company.Product.ApplicationCore`

#### 判斷準則

> 如果拿掉 WinForm、資料庫、Serilog，這層還能完整存在嗎？

### Infrastructure

#### 慣例語意

- 技術實作層
- 可替換、但集中管理
- 對 Core 提供支援能力

#### 通常包含

- Logging
- Persistence（EF、SQLite）
- Messaging
- Cache
- Configuration
- FileSystem
- 外部系統整合

#### 特性

- 高技術耦合
- 跨模組共用
- 統一初始化與生命週期

#### 命名範例

- `Company.Product.Infrastructure`
- `Company.Product.Infrastructure.Logging`
- `Company.Product.Infrastructure.Persistence`

#### 判斷準則

> 這段程式碼是否「**如果換技術，就整包換**」？

### Application

#### 慣例語意

- 使用案例層
- 業務流程協調
- 連接 UI 與 Core

#### 通常包含

- Use Case
- Application Service
- Transaction Boundary
- DTO 組裝

#### 不應包含

- UI 控制項
- 技術細節（EF、Serilog）

#### 命名範例

- `Company.Product.Application`
- `Company.Product.AppServices`

### Presentation / UI

#### 慣例語意

- 使用者互動
- 展示層

#### 通常包含

- WinForm / WPF / Web
- ViewModel
- Controller

#### 命名範例

- `Company.Product.WinForm`
- `Company.Product.UI`

## 二、Framework / Shared / Common 命名的真相

### Framework

#### 業界實際用法

- 內部平台
- 多產品共用
- 比單一 App 壽命更長

#### 何時該用

- 有 2+ 專案共用
- 有版本控管
- 有相容性考量

#### 風險

- 容易變成垃圾場
- 必須有嚴格責任邊界

### Common / Shared

#### 慣例語意

- 雜項
- 快速重用

#### 強烈警告

- 一旦出現，架構開始腐化
- 不適合你現在這套 Logging

## 三、命名層級的決策表（快速判斷）

- 是否包含業務規則？→ Core
- 是否負責技術整合？→ Infrastructure
- 是否協調流程？→ Application
- 是否處理顯示？→ UI
- 是否跨多產品？→ Framework

## 一句話總結

**Core 是「為什麼做」，Infrastructure 是「怎麼做到」。**

# 已做成 NuGet 是否可稱為 Framework

**可以，但不必然。**
是否命名為 Framework，取決於「責任與影響範圍」，而不是是否發佈成 NuGet。

## Framework 的判定標準

符合以下多數條件，才適合稱為 Framework：

- 提供一套「使用方式與結構約束」
- 對 App 或模組有主導性，而非被動被呼叫
- 涵蓋多個橫切關注點
- 會影響專案的啟動流程或生命週期
- 多個系統或產品線共用

## NuGet 常見但不一定是 Framework 的類型

- Utilities / Helpers
- Extension Methods
- 單一技術封裝（Serial、Socket、File）
- 純 Abstractions / Contracts
- 單一功能 Library

這類即使發佈成 NuGet，通常仍只是「Library」，不建議直接叫 Framework。

## 建議的實務命名分類

### Framework.*

- 適用於平台級、結構級套件
- 例：DI、Logging、Module Loader、Task Orchestration
- 命名範例
    - Xxx.Framework
    - Xxx.Framework.Logging
    - Xxx.Framework.Bootstrap

### Infrastructure.*

- 技術實作型套件
- 例：EF、Dapper、SerialPort、Camera SDK
- 命名範例
    - Xxx.Infrastructure.Sql
    - Xxx.Infrastructure.Serial
    - Xxx.Infrastructure.Camera

### Abstractions / Contracts

- 穩定介面與資料契約
- 例：ILogger、IClock、DTO
- 命名範例
    - Xxx.Abstractions
    - Xxx.Contracts

### Application.*

- 使用案例層
- 例：Workflow、UseCase、Coordinator
- 命名範例
    - Xxx.Application
    - Xxx.Application.ScrewLock

### Common / Utilities（慎用）

- 無業務語意的輔助工具
- 僅限真正跨所有層的功能
- 命名範例
    - Xxx.Common
    - Xxx.Utilities

## 判斷 NuGet 應該放哪一類的快速檢查

- 是否定義「你應該怎麼寫程式」？
    - 是 → Framework
- 是否只是「幫你把某技術包好」？
    - 是 → Infrastructure
- 是否只放 Interface / DTO？
    - 是 → Abstractions / Contracts
- 是否與某個業務流程強耦合？
    - 是 → Application
- 是否沒有清楚責任？
    - 是 → 該重構或拆分

## 重構整理時的實務策略

- NuGet 名稱先不要急著改
- 專案內 Namespace 先對齊責任
- Framework 與 Infrastructure 明確切開
- Common 若超過 1~2 個責任，立即拆

## 簡短結論

- NuGet ≠ Framework
- Framework 是「平台級約束與能力集合」
- Infrastructure 是「技術實作」
- Abstractions 是「穩定邊界」
- 命名一旦清楚，專案整理成本會快速下降
