# FakeSerialPortService

我會用「是否會讓 Modbus 測試產生錯覺」這個標準來做嚴格 Review，而不是只看程式碼風格。

整體結論先說：

✔ 架構方向正確
✔ 沒有破壞協定純度
✔ 沒有 Fake 過度聰明問題
⚠ 但有幾個會影響工控等級測試嚴謹性的細節需要修正

下面逐點分析。

## 一、設計層級判斷

你的 FakeSerialPortService 是：

傳輸層 Fake
不理解 Modbus
不自動補 CRC
不解析 Function Code
不生成回應

這點是完全正確的。

它只模擬：

- 狀態
- 傳送
- 接收
- 錯誤
- 驗證旗標

沒有越界到協定層。

這是乾淨設計。

## 二、重要優點確認

### 1️⃣ SendDataAsync 會複製 buffer

這是正確做法：

```csharp
var copy = new byte[count];
Buffer.BlockCopy(...)
```

避免測試端修改原 buffer 導致隱性錯誤。

這是專業細節。

### 2️⃣ 不自動設定 IsTransmissionVerified

這點非常重要。

Fake 不幫你決定通訊是否成功，
由測試端控制。

這避免測試永遠成功。

設計正確。

### 3️⃣ StateChanged 事件有鎖保護

事件引用抓出來再 invoke。

避免死鎖與 race condition。

是正確實作方式。

## 三、真正需要修正的地方

以下是嚴格模式下的問題。

### 問題 1：SimulateReceiveData 沒有複製 buffer

目前：

```csharp
var args = new DataReceivedEventArgs(..., data, true);
```

如果測試端在呼叫後修改 data 內容，

事件接收端會看到被改動的內容。

這在實際 SerialPort 中不會發生。

實際 SerialPort 會給你一份獨立 buffer。

建議改為：

```csharp
var copy = new byte[data.Length];
Buffer.BlockCopy(data, 0, copy, 0, data.Length);
```

否則這是隱性錯誤來源。

這是必修。

### 問題 2：沒有模擬分段接收能力

目前：

SimulateReceiveData 是一次性丟完整資料。

但在 Modbus RTU 實際場景會出現：

- 半包
- 黏包
- 多次 DataReceived 事件

你目前只能：

一次送完整 frame

這會讓 Modbus Parser 測試不完整。

建議新增：

```csharp
SimulateReceiveFragments(byte[][] fragments)
```

或

```csharp
SimulateReceiveData(byte[] data, int fragmentSize)
```

否則你的整合測試不是真實 RTU 行為。

這是重要改進點。

### 問題 3：沒有延遲能力

真實 SerialPort：

- DataReceived 不是立即觸發
- 可能晚幾 ms
- 可能在 ThreadPool 執行

現在你的 Fake 是同步呼叫。

這會讓：

- Timeout 測試失真
- Race condition 測試失真

建議增加：

```csharp
public int SimulatedReceiveDelayMs { get; set; }
```

並用 Task.Run + Delay 模擬。

這會讓測試更接近實機。

### 問題 4：OpenAsync 沒檢查 cancellationToken

你忽略 cancellationToken。

在工控長期運行架構下，

Open/Send 都應尊重 cancellationToken。

建議：

如果 cancellationToken.IsCancellationRequested
回傳取消結果。

雖然是 Fake，但應符合契約。

### 問題 5：IsOpen 定義可能不嚴謹

目前：

```csharp
IsOpen => State != Disconnected
```

當 State = Fault 時 IsOpen = true。

真實 SerialPort 發生 Fault 後通常視為不可用。

你要確認這是否符合你系統語意。

這不是錯誤，但需確認語意一致性。

## 四、是否會破壞 Modbus 測試純度？

答案：不會。

因為：

- 沒有解析 Modbus
- 沒有自動生成回應
- 沒有驗證 CRC
- 沒有理解 Function Code

Fake 是純線路層。

這是正確的。

## 五、是否需要 Modbus 專屬 Mock？

在看到這份實作後，

結論更加確定：

不需要。

這個 FakeSerialPortService 已經是正確抽象層。

如果再做 FakeModbusTransport，

反而會破壞測試可信度。
