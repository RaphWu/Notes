# 禁止 `async void`、禁止 fire-and-forget

這個規則對企業級工控通訊框架非常重要，尤其是 SerialPort 背景任務、Channel、心跳、傳輸驗證，都涉及多 Task 和非同步操作。必須 **明確禁止**，避免未捕捉例外導致背景任務崩潰。

---

# 24/7 WinForms 工控最低必要安全清單

目標：
在不過度工程化的前提下，確保系統可長時間穩定運行（>30 天）且可快速定位問題。

以下為「最低必要等級」，非理想完美版，而是實務可落地版本。

## 一、資源安全層（最高優先）

### 1. 嚴格 IDisposable 規範

所有以下類型必須 using 或手動 Dispose：

1. Bitmap
2. Graphics
3. Pen
4. Brush
5. Font
6. Image
7. Icon
8. Region

禁止在 OnPaint 中 new 未釋放物件。

### 2. Image 置換必釋放舊物件

```csharp
var old = pictureBox.Image;
pictureBox.Image = newBitmap;
old?.Dispose();
```

否則 24 小時內 GDI 會線性成長。

### 3. 禁止頻繁 new Control

動態產生 Control 必須：

1. Controls.Remove
2. Dispose

不可只 Remove。

### 4. GDI / Handle 監控（必做）

每 5 秒紀錄：

```csharp
Process.GetCurrentProcess().HandleCount
Process.GetCurrentProcess().PrivateMemorySize64
```

判斷原則：

1. 波動 → 正常
2. 緩慢線性成長 → Leak

不做監控 = 無法診斷長期故障。

## 二、UI Thread 保護層（工控必要）

### 5. 強制 UI Thread 操作

所有跨執行緒 UI 更新：

```csharp
if (InvokeRequired)
    Invoke(...)
```

禁止直接操作 UI。

### 6. UI Freeze Watchdog

每 3 秒檢查：

1. UI 是否回應
2. 是否超過 5 秒未處理訊息

若卡死：

1. 記錄 Log
2. 可選擇自動重啟

24/7 系統不能容忍 UI 無回應。

## 三、例外隔離層（避免整體崩潰）

### 7. Application 層級例外捕捉

```csharp
Application.ThreadException
AppDomain.CurrentDomain.UnhandledException
```

處理策略：

1. 記錄錯誤
2. 關閉當前畫面
3. 不讓整個系統崩潰

工控環境首要原則：不中斷主程序。

### 8. 子功能隔離

即時影像 / 通訊 / 資料處理：

1. 例外不可往 UI Thread 冒泡
2. 任務失敗自動重啟
3. 不可阻塞主 UI

## 四、視窗生命週期管理（簡化版）

### 9. 禁止重複開窗

每個工具視窗：

1. 單例
2. 已存在則 BringToFront

避免 10 個隱藏視窗累積 Handle。

### 10. Close = Dispose

若非 Persistent 視窗：

```csharp
form.Close();
form.Dispose();
```

避免殭屍 Handle。

## 五、記憶體安全策略

### 11. 強制使用 x64

禁止 32bit 工控部署。

原因：

1. 32bit 記憶體上限 ~1.5GB
2. 長時間碎片化會崩潰

### 12. 定期 GC 不是解法

不要用：

```csharp
GC.Collect();
```

解決 Leak。

如果需要手動 GC 才能穩定 = 架構有問題。

## 六、通訊與背景任務安全

### 13. 所有長時間任務必須

1. 可取消（CancellationToken）
2. 有 Timeout
3. 不阻塞 UI
4. 失敗可重啟

### 14. 禁止無限 while(true)

必須：

1. 有延遲
2. 有例外捕捉
3. 有退出條件

## 七、日誌與追蹤（不可省略）

### 15. Log 最少包含

1. 啟動時間
2. 異常
3. 記憶體使用量
4. GDI 數量
5. 通訊失敗

沒有 Log = 事後無法還原事故。

## 八、不要做的事

以下不是最低必要：

1. 過度複雜 FormManager 框架
2. 反射式畫面生成
3. Attribute 權限控制
4. 過度插件化
5. 自製 UI Framework

工控系統首要是穩定，不是架構炫技。

## 九、最低必要架構模型

實務穩定組合：

1. 簡化版 FormManager（單例控制）
2. GDI / Memory 監控
3. UI Watchdog
4. 全域例外捕捉
5. 嚴格 Dispose 規範
6. x64 發佈

做到這六項，
已足夠支撐 24/7 生產線系統。

## 十、風險排序（重要）

若時間有限，優先順序：

1. x64
2. GDI 監控
3. Dispose 規範
4. 例外隔離
5. UI Watchdog
6. 視窗管理

自訂繪圖專案最常死於：

GDI 緩慢上升 → Handle 用盡 → 畫面無法建立 → 崩潰。

# 自訂繪圖專案專用的 24/7 WinForms 工控框架模板

- FormManager（簡化版，單例管理 + Dispose）
- GDI / Handle / Memory 監控
- UI Freeze 偵測
- 全域例外隔離

以下是一個可直接落地的範例框架：

## 一、FormManager（簡化版）

```csharp
public static class FormManager
{
    private static readonly Dictionary<string, Form> _forms = new Dictionary<string, Form>();
    private static readonly object _lock = new object();

    public static T ShowForm<T>(bool persistent = false) where T : Form, new()
    {
        lock (_lock)
        {
            string key = typeof(T).FullName;

            if (_forms.ContainsKey(key))
            {
                var existing = _forms[key];
                if (persistent)
                {
                    existing.Show();
                    existing.BringToFront();
                    return (T)existing;
                }
                else
                {
                    existing.Close();
                    _forms.Remove(key);
                }
            }

            var form = new T();
            if (!persistent)
            {
                form.FormClosed += (s, e) =>
                {
                    form.Dispose();
                    _forms.Remove(key);
                };
            }

            _forms[key] = form;
            form.Show();
            return form;
        }
    }

    public static void CloseForm<T>() where T : Form
    {
        lock (_lock)
        {
            string key = typeof(T).FullName;
            if (_forms.TryGetValue(key, out var form))
            {
                form.Close();
            }
        }
    }
}
```

## 二、Safe UI Invoker

```csharp
public static class SafeUIInvoker
{
    public static void Invoke(Action action)
    {
        if (action == null) return;
        if (System.Windows.Forms.Application.OpenForms.Count == 0) return;

        var form = System.Windows.Forms.Application.OpenForms[0];
        if (form.InvokeRequired)
            form.Invoke(action);
        else
            action();
    }
}
```

## 三、UI Freeze 監控

```csharp
public static class UIFreezeMonitor
{
    private static System.Threading.Timer _timer;
    private static DateTime _lastUiResponse;
    private static int _timeoutMs = 5000;
    public static Action OnFreezeDetected;

    public static void Start(int checkIntervalMs = 1000)
    {
        _lastUiResponse = DateTime.UtcNow;
        _timer = new System.Threading.Timer(_ =>
        {
            SafeUIInvoker.Invoke(() => _lastUiResponse = DateTime.UtcNow);

            if ((DateTime.UtcNow - _lastUiResponse).TotalMilliseconds > _timeoutMs)
                OnFreezeDetected?.Invoke();

        }, null, 0, checkIntervalMs);
    }

    public static void Stop() => _timer?.Dispose();
}
```

## 四、資源監控（GDI / Handle / Memory）

```csharp
using System.Diagnostics;
using System.Runtime.InteropServices;

public static class ResourceMonitor
{
    [DllImport("user32.dll")]
    private static extern int GetGuiResources(IntPtr hProcess, int uiFlags);

    private const int GR_GDIOBJECTS = 0;
    private const int GR_USEROBJECTS = 1;

    public static int GetGdiCount() => GetGuiResources(Process.GetCurrentProcess().Handle, GR_GDIOBJECTS);
    public static int GetUserHandleCount() => GetGuiResources(Process.GetCurrentProcess().Handle, GR_USEROBJECTS);
    public static int GetHandleCount() => Process.GetCurrentProcess().HandleCount;
    public static long GetPrivateMemoryMB() => Process.GetCurrentProcess().PrivateMemorySize64 / 1024 / 1024;
}
```

## 五、穩定性監控整合

```csharp
public static class StabilityMonitor
{
    private static System.Threading.Timer _timer;
    public static Action<string> OnWarning;
    public static Action<string> OnCritical;

    public static void Start(int intervalMs = 10000)
    {
        _timer = new System.Threading.Timer(_ => Check(), null, 0, intervalMs);
    }

    private static void Check()
    {
        var gdi = ResourceMonitor.GetGdiCount();
        var handle = ResourceMonitor.GetHandleCount();
        var memory = ResourceMonitor.GetPrivateMemoryMB();

        if (gdi > 8000) OnCritical?.Invoke($"GDI HIGH: {gdi}");
        else if (gdi > 5000) OnWarning?.Invoke($"GDI WARNING: {gdi}");

        if (handle > 10000) OnCritical?.Invoke($"HANDLE HIGH: {handle}");
        if (memory > 1500) OnCritical?.Invoke($"MEMORY HIGH: {memory} MB");
    }

    public static void Stop() => _timer?.Dispose();
}
```

## 六、全域例外隔離

```csharp
public static class ExceptionHandler
{
    public static void Register()
    {
        Application.ThreadException += (s, e) =>
        {
            LogError("ThreadException", e.Exception);
        };

        AppDomain.CurrentDomain.UnhandledException += (s, e) =>
        {
            LogError("UnhandledException", e.ExceptionObject as Exception);
        };
    }

    private static void LogError(string type, Exception ex)
    {
        // 這裡可寫 Log 或顯示警告
        Debug.WriteLine($"{type}: {ex}");
    }
}
```

## 七、主程式整合範例

```csharp
static class Program
{
    [STAThread]
    static void Main()
    {
        ExceptionHandler.Register();
        Application.EnableVisualStyles();
        Application.SetCompatibleTextRenderingDefault(false);

        UIFreezeMonitor.Start();
        StabilityMonitor.Start();

        UIFreezeMonitor.OnFreezeDetected = () =>
        {
            Debug.WriteLine("UI Freeze Detected!");
        };

        StabilityMonitor.OnCritical = msg =>
        {
            Debug.WriteLine($"Critical: {msg}");
        };

        Application.Run(new MainForm());
    }
}
```

### 特點

1. 適合自訂繪圖 / 即時曲線 / 即時影像
2. 24/7 長時間運行，UI Freeze、GDI、Memory 有監控
3. 簡化 FormManager 保持單例 / Dispose
4. 全域例外捕捉避免主程式崩潰
5. 可在現場非工程師操作下穩定運行

# 在「低階 / 即時 / 設備控制」環境中， CancellationToken 使用注意事項

在「低階 / 即時 / 設備控制」環境中，CancellationToken 不是不能用，而是「不能照一般 .NET 寫法用」。

## 避免用法

### ❌ 1. 用 Exception 當 cancellation（最該避免）

```csharp
cancellationToken.ThrowIfCancellationRequested();
```

或

```csharp
catch (OperationCanceledException)
```

#### 為什麼？

- Exception 成本高（stack unwind）
- timing 不可預測（對即時系統不好）
- 會造成 jitter（控制系統很敏感）

👉 在設備控制 / PLC / motion control：

> ❌ **Exception = 不可控延遲來源**

### ❌ 2. 把 cancellationToken 傳進 `Task.Delay`

```csharp
await Task.Delay(10, cancellationToken);
```

#### 問題

- cancellation 時會 **丟 exception**
- 你又回到問題 1
- 在 tight loop 中會產生不穩定行為

### ❌ 3. 完全依賴 async/await 控制流程

在低階系統：

- async scheduling ≠ deterministic timing
- thread pool ≠ real-time safe

👉 特別是：

- 馬達控制
- 感測器 polling
- IO cycle

## ✅ 在低階系統「推薦」的做法

### ✔️ 原則：**Cooperative flag（協作式中斷）**

👉 用 flag，不用 exception

### 🔹 寫法（推薦）

```csharp
while (!isReady)
{
    if (cancellationToken.IsCancellationRequested)
        return false;

    await Task.Delay(10); // ❗ 不帶 token

    isReady = CheckSensor();
}
```

### 🧠 為什麼這樣比較好？

| 項目    | Exception-based | Flag-based |
| ------- | --------------- | ---------- |
| latency | 不可預測        | ✅ 可控    |
| CPU     | 普通            | 普通       |
| jitter  | ❌ 有           | ✅ 低      |
| 可讀性  | 普通            | ✅ 清楚    |
| RT safe | ❌              | ⚠️ 較安全  |

## ⚠️ 進階：更低階（更接近設備）

如果你真的在做：

- 運動控制
- 即時控制
- 高頻 polling（<10ms）

👉 甚至 **連 `Task.Delay` 都該避免**

### 🔥 改用

#### ✔️ `Stopwatch + Spin / Sleep`

```csharp
var sw = Stopwatch.StartNew();

while (!isReady)
{
    if (cancellationToken.IsCancellationRequested)
        return false;

    if (sw.ElapsedMilliseconds >= 10)
    {
        sw.Restart();
        isReady = CheckSensor();
    }

    Thread.SpinWait(50); // 或 Thread.Sleep(1)
}
```

## 🚨 超重要（你這個場景）

你這段其實是：

👉 **機台 + 感測器 + 旋轉**

這種屬於：

> ⚙️ **soft real-time system（軟即時系統）**

### 在這種系統中

#### ✔️ cancellation 應該

- 是「狀態」不是「例外」
- 是「控制訊號」不是「錯誤」

---

## ✅ 最推薦模式（設備控制版）

```csharp
while (isSearching)
{
    if (cancellationToken.IsCancellationRequested)
    {
        StopMotor();   // 很重要
        return false;
    }

    if (CheckSensor())
    {
        StopMotor();
        return true;
    }

    Thread.Sleep(10); // 或更精細控制
}
```

## 🧾 總結（很關鍵）

### ❌ 避免

- ThrowIfCancellationRequested
- OperationCanceledException flow
- Task.Delay(token)
- 過度 async/await 控制 loop

### ✅ 建議

- 用 `IsCancellationRequested`
- 用 flag 控制流程
- cancellation = 狀態，不是例外
- 搭配明確 cleanup（Stop motor）

## 🚀 一句話結論

👉 **在低階/設備控制中：**

> 用 `CancellationToken` 的「flag」，不要用它的「exception」
