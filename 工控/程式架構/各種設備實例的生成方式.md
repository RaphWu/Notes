# 在工控（Industrial Control）系統中，設備實例的非 DI 生成方式

在沒有使用 DI Container（如 Autofac）的前提下，生成方式會直接影響**可測試性、擴充性、錯誤隔離能力與生命週期管理**。
主要有兩種策略：

1. 直接 `new`
2. 使用工廠（Factory Pattern）

以下是場景分析。

## 何時適合直接 `new`

### 適用條件

1. 類型固定，無需動態切換實作
2. 不存在複雜建構參數
3. 不需要抽換為 Fake / Mock
4. 專案規模小，設備種類單一
5. 不涉及設備自動重建或重連策略

### 典型場景

1. 單一 RS-232 設備
2. 固定協議（例如固定為 ASCII）
3. 不需模擬測試

### 優點

1. 結構單純
2. 可讀性高
3. 除錯直接

### 缺點

1. 產生強耦合
2. 未來擴充時需修改呼叫端
3. 難以做自動化測試

### 範例

```csharp
var device = new DlRs1ADevice(config);
device.Connect();
```

這種方式在小型專案或 POC 階段完全合理。

## 何時適合使用工廠（Factory）

當設備生成邏輯「不只是單純 new」，就應該使用工廠。

### 適用條件

1. 多設備類型（RS232 / TCP / Modbus）
2. 依設定檔決定實作類型
3. 需要包裝重連機制
4. 需要在不同模式下切換 Fake/Real
5. 建構過程涉及多層依賴

### 典型場景

1. 同一控制軟體支援不同品牌設備
2. 工廠端會依 ModelType 決定實例
3. 需要統一初始化流程（例如驗證通訊）

### 優點

1. 呼叫端解耦
2. 容易擴充
3. 易於集中管理生命週期
4. 可統一加入 logging、retry、state machine

### 缺點

1. 多一層抽象
2. 結構稍複雜

### 範例

```csharp
var device = DeviceFactory.Create(config);
device.Connect();
```

或

```csharp
IDevice device = _deviceFactory.Create(modelType);
```

## 工控實務建議（重點）

在工控環境下，我會建議：

1. 單一設備專案 → 可直接 new
2. 有可能擴充型號 → 一律使用工廠
3. 需要長時間穩定運行（24/7）→ 使用工廠集中管理
4. 有 Fake 測試需求 → 一定用工廠

## 架構層級建議（以你常見的分層為例）

若你有這種結構：

Application
↓
Comm Layer
↓
Transport
↓
IO

那麼建議：

1. Application 不要直接 new Comm 實作
2. 由 CommFactory 負責生成
3. Transport 可由工廠內部 new（除非也要抽換）

這樣未來就算改成 DI，也只是把工廠邏輯搬進 Container 註冊。

## 進階建議（工控專用思維）

若系統具備：

1. 自動重連
2. Faulted 狀態恢復
3. 通訊驗證（IsTransmissionVerified）
4. 生命週期狀態機

那麼：

設備的生成就不只是 new，而是「初始化流程的一部分」，這時候一定要工廠，甚至是 Builder + Factory。

## 簡單決策表

1. 小型單設備專案 → new
2. 多型號或未來可能擴充 → Factory
3. 需 Fake 或測試 → Factory
4. 有狀態機或重連邏輯 → Factory
5. 長期維運工廠產線系統 → Factory（強烈建議）

## 一句結論

在工控領域，只要不是一次性小工具，**幾乎都應該用工廠，而不是直接 new**。

# 工控設備實例生成：`new` vs Factory vs DI 完整比較

在工控系統（24/7、長時間穩定運作、需可恢復性）的架構設計中，實例生成策略其實影響的是：

1. 依賴管理能力
2. 狀態生命週期控制
3. 可測試性
4. 維運成本
5. 未來擴充彈性

以下做精準工程向比較。

## 三種方式本質差異

### 1️⃣ 直接 `new`

特性：

1. 呼叫端負責建構
2. 強耦合
3. 依賴固定寫死

本質：
「物件生成權在使用者手上」

### 2️⃣ Factory

特性：

1. 生成邏輯集中
2. 呼叫端只依賴抽象
3. 可封裝初始化流程

本質：
「物件生成權在你設計的架構內部」

### 3️⃣ DI（例如 Autofac）

特性：

1. 生成權交給 Container
2. 支援生命週期管理（Singleton/Scoped/Transient）
3. 支援依賴圖自動建構

本質：
「物件生成權在容器」

## 架構維度比較（工控角度）

### 1. 耦合度

new：

1. 高耦合
2. 呼叫端知道具體類型

Factory：

1. 中低耦合
2. 呼叫端只知道抽象

DI：

1. 最低耦合
2. 呼叫端只宣告需求

結論：DI 最乾淨

### 2. 多設備擴充能力

new：

1. 每新增型號就要改呼叫端
2. 開放封閉原則破壞

Factory：

1. 只修改 Factory
2. 呼叫端不動

DI：

1. 只修改註冊
2. 呼叫端完全不動

結論：Factory ≈ DI（DI 更乾淨）

### 3. 生命週期管理（工控關鍵）

工控常見需求：

1. 設備 Singleton
2. 通訊層長時間保持
3. Faulted 需重新建立

new：

1. 呼叫端自行管理
2. 容易散落各處

Factory：

1. 可集中管理
2. 可加入重建策略

DI：

1. 容器可控生命週期
2. 但不適合頻繁重建硬體物件

⚠ 工控重要觀念：
硬體設備物件不適合頻繁由 DI Container 建立與銷毀。
原因：

1. Container 不理解硬體狀態
2. 容器生命週期 ≠ 設備生命週期

結論：
工控設備本體通常不建議完全交給 DI 控制生命週期。

### 4. 測試能力

new：

1. 難以替換 Fake
2. 幾乎無法做單元測試

Factory：

1. 可注入 FakeFactory
2. 或讓 Factory 支援模式切換

DI：

1. 最強
2. 測試時替換註冊即可

結論：DI 最優

### 5. 複雜初始化流程

若設備建立流程包含：

1. 連線
2. 驗證傳輸（IsTransmissionVerified）
3. 狀態機啟動
4. 事件綁定

new：

1. 呼叫端需自己寫流程
2. 容易分散

Factory：

1. 可封裝為 CreateAndInitialize
2. 初始化流程集中

DI：

1. 不適合做複雜初始化流程
2. Container 不應承擔業務初始化邏輯

⚠ 重要：
DI 是依賴管理工具，不是初始化流程管理工具。

## 工控實務結論（精準版本）

### 小型工具型專案

直接 new 即可

### 單設備 + 長時間穩定運行

使用 Factory（最佳平衡）

### 多設備 + 企業級系統

建議：

DI 管理「服務層」
Factory 管理「硬體設備層」

## 工控最佳實務架構（推薦）

Container（例如 Autofac）負責：

1. Application Service
2. Domain Service
3. Log / Config
4. Factory 本身

Factory 負責：

1. new 設備
2. 初始化
3. 狀態驗證
4. 重建策略

設備物件：
不直接註冊為 Singleton 交給 DI

## 一句話總結

DI 是依賴管理工具
Factory 是硬體生成控制工具

在工控領域：
「設備實例用 Factory 管」
「服務層用 DI 管」

## 給你一個架構層級判斷公式

如果這個物件：

1. 有硬體資源
2. 有連線狀態
3. 有故障恢復邏輯
4. 可能被重新建立

那麼：
不要讓 DI 直接掌控它的生命週期。
