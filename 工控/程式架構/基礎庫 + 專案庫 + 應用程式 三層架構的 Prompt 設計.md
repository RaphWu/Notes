# 基礎庫 + 專案庫 + 應用程式 三層架構的 Prompt 設計

重點是把「架構意圖」寫進 Prompt，而不是只說「幫我重構」。
以下直接給你可套用的設計方式。

## 新設計（父類 + 子類繼承）時，Prompt 怎麼寫

建議用 6 段式 Prompt，每段都要具體：

1. 背景/上下文
   專案類型、目標框架（比如 net48）、既有類別命名風格。
2. 父類責任（一定要明確）
   例如生命週期、共用驗證、共用錯誤處理、事件管線。
3. 子類責任（一定要明確）
   例如機台型號限制（366）、通道/Port 限制、覆寫規則。
4. 擴充點規格
   哪些要 abstract、哪些 virtual、哪些要 sealed override。
5. 輸出格式要求
   要求產出：類別圖、檔案清單、完整程式碼、相依關係、測試案例。
6. 驗收標準
   例如：不破壞 IDaqService、可替換注入、現有呼叫端不改或最小改動。

### 新設計 Prompt 範本（可直接貼給 AI）

請協助我設計 USB-4704 服務架構，採「父類 + 子類」繼承。

- 技術限制：C#、net48、維持現有 DI 與 logging 風格。
- 目標：將共用行為放在父類，機台特化邏輯放在子類。
- 父類要包含：Initialize/Dispose、Polling 啟停流程、例外與日誌、事件拋送共用機制。
- 子類（例如 Calin.ScrewFastening.Services.DaqService）要包含：366 機台專屬 AI/DI/DO 限制、通道驗證、專屬診斷欄位。
- 請定義：
    - 父類中 abstract / virtual / sealed 方法清單與理由
    - 子類可覆寫與不可覆寫邊界
    - 建議檔案結構與命名
- 請輸出：
    - 架構說明
    - 類別責任表
    - 完整可編譯程式碼
    - 最小單元測試清單
    - 風險與替代方案。

## 舊專案改成三層架構時，Prompt 怎麼寫

舊專案改造要改成 分階段 Prompt，一次叫 AI 全改通常風險高。
建議 4 個回合：

1. 盤點回合（只分析不改碼）
   找重複邏輯、耦合點、子類特化點。
2. 抽象設計回合（先出設計）
   先出父類 API 與遷移地圖。
3. 遷移實作回合（分檔改）
   一次只改一個類別，附差異說明。
4. 相容驗證回合（收斂）
   驗證公開介面、事件、例外、執行序安全不破壞。

### 舊專案重構 Prompt 範本（可直接貼給 AI）

我要把現有 DAQ 服務重構成「父類 + 子類」繼承架構，請用「先分析、後修改」流程。

第 1 階段（不要改碼）：
- 盤點目前 DaqService 中哪些是共用邏輯，哪些是 366 特化邏輯。
- 列出候選父類方法、欄位、事件，與保留在子類的方法。
- 提供風險（行為改變、執行緒、初始化順序、Dispose）與遷移順序。
第 2 階段（產出設計）：
- 定義父類 BaseDaqService（名稱可調整）與子類 DaqService 的責任邊界。
- 保證 IDaqService 相容，既有呼叫端最小改動。
- 提供完整檔案異動清單。
第 3 階段（實作）：
- 依檔案逐一提供可編譯修改內容。
- 每個檔案附「為何這樣改」與「不改會有何問題」。
第 4 階段（驗證）：
- 提供回歸測試清單（初始化、Polling、事件、異常、Dispose、重連）。
- 提供手動驗證步驟與預期結果。

## 實戰建議（精簡）

- Prompt 一定要有「非目標」：例如「不要改 public API 名稱」。
- Prompt 一定要有「驗收條件」：例如「編譯通過 + 既有測試通過 + 行為一致」。
- 先讓 AI 產生「設計草案」，確認後再要「完整改碼」。

# 使用方式

分次貼、按階段互動（不要一次把四個階段整個丟進去）。
原因：分階段能降低風險、方便審查、每步都有可驗收的成果，且較易回溯與修正。

## 簡要工作流程（每階段一次互動）

- 階段 1 — 盤點（貼上欲重構的原始檔案/相關檔案）
    - 要求：列出「共用邏輯 / 特化邏輯」、「風險點」與「遷移順序」。
    - 驗收：你確認盤點無遺漏才進入下一階段。
- 階段 2 — 抽象設計（貼上或接受父類 API 草案）
    - 要求：父類/子類責任、abstract/virtual 決策、檔案清單、類別圖。
    - 驗收：你核准設計或要求修改。
- 階段 3 — 分檔實作（每次貼一個檔案的原始內容或 patch）
    - 要求：回傳可編譯的完整檔案修改、差異說明、改動理由、測試建議。
    - 驗收：在你的本地編譯/測試通過後才繼續下一檔。
- 階段 4 — 驗證與回歸（貼上測試清單與 CI/驗收步驟）
    - 要求：回歸測試步驟、手動驗證要點、可能的回退方案。
    - 驗收：全部測試通過、行為與既有 API 相容。

## 為什麼要分次

- 每階段輸出較小、容易審閱與回饋。
- 可在實作前攔截設計錯誤（避免大量重工）。
- 方便在 CI / 本地測試失敗時快速回溯到某一階段。

## 操作提示（要貼給 AI 的最簡短版）

- 階段 1 範例：「請分析並盤點此檔案（附上 DaqService.cs），列出可抽象到父類的成員、366 專屬部分、重大風險與建議遷移順序。」
- 階段 2 範例：「根據盤點結果，產出 BaseDaqService 設計（methods/properties/events），並列出哪些為 abstract/virtual/sealed。」
