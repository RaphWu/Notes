# RBAC、PBAC 與 ABAC 的優缺點與適合場合

## 概念比較

1. **RBAC（Role-Based Access Control，角色基礎存取控制）**
    
    - 核心思想：**使用者的權限由角色決定**。
    - 角色與權限是固定關聯，使用者被分配角色即擁有該角色的所有權限。
    - 優點是簡單直觀、易於管理。

2. **PBAC（Policy-Based Access Control，政策基礎存取控制）**
    
    - 核心思想：**存取決策依據集中管理的政策（Policy）規則**。
    - 策略可依角色、屬性、時間、環境等條件決定是否允許存取。
    - 強調集中式管理與策略可擴展性。

3. **ABAC（Attribute-Based Access Control，屬性基礎存取控制）**
    
    - 核心思想：**存取決策依據使用者、資源、環境等屬性**。
    - 支持細粒度、動態授權，策略通常以屬性條件形式表達。

> 可以理解為 RBAC 是粗粒度控制，ABAC 是細粒度控制，PBAC 則偏向策略集中化管理，ABAC 與 PBAC 常有交集。

## 優缺點比較

1. **RBAC**
    - 優點：
        - 簡單易懂、易管理
        - 適合角色明確、權限固定的系統
        - 實現成本低
    - 缺點：
        - 粒度粗，難以處理動態條件
        - 當角色過多或組合複雜時易混亂
        - 不適合需依屬性判斷的場景

2. **PBAC**
    - 優點：
        - 政策集中管理，可修改政策無需改程式碼
        - 支援複雜條件與多種屬性
        - 適合大型系統和多樣化需求
    - 缺點：
        - 策略設計與管理複雜
        - 決策性能受策略數量影響
        - 單純使用不易快速判斷簡單權限

3. **ABAC**
    - 優點：
        - 支持精細化控制和動態授權
        - 可以根據多種屬性組合做決策
        - 高度可擴展，適合複雜業務場景
    - 缺點：
        - 屬性定義與管理成本高
        - 單獨使用時策略可能難以直觀管理
        - 需要策略引擎或框架輔助

## 單獨使用的適合場合

1. **RBAC**
    - 系統角色清楚且權限變化少
    - 例如：企業內部管理系統、簡單流程控制

2. **PBAC**
    - 系統規則複雜，需集中策略管理
    - 例如：大型企業策略管理、合規要求高的系統

3. **ABAC**
    
    - 需要精細控制且存取依屬性判斷
    - 例如：金融、醫療、雲端多租戶平台

## 合併使用的適合場合

1. **RBAC + ABAC**
    - RBAC：先做粗粒度篩選，決定使用者角色可存取的範圍
    - ABAC：再做細粒度條件控制，例如根據文件敏感度或存取時間動態授權
    - 適合大型企業系統或多租戶平台

2. **PBAC + ABAC**
    - PBAC：集中管理政策模板與決策流程
    - ABAC：提供屬性條件判斷，支援動態授權
    - 適合複雜規則與動態條件並存的高安全系統

## 思路建議

1. 系統簡單 → **RBAC 足夠**
2. 系統需要動態授權 → **ABAC**
3. 大型系統或複雜規則 → **RBAC + ABAC 或 PBAC + ABAC**
4. 建議先用 RBAC 做角色篩選，再用 ABAC 做細粒度決策
5. PBAC 可用於策略集中化管理，增加可維護性
