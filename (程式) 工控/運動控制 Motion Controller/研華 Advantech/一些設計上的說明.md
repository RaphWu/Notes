# 參數持久化 (Chat-GPT)

**參數持久化如果放錯層，會直接污染核心架構。**

我直接給結論，然後說明原因與最佳設計。

## 結論（Level 5 正確位置）

### ❌ 不應放在核心層 `Calin.MotionControl.Advantech`

### ❌ 不應直接寫在 PCI / EtherCAT 子專案

### ✅ 應新增「獨立的組態/持久化層」

建議新增：

```csharp
Calin.MotionControl.Advantech.Configuration
```

或

```csharp
Calin.MotionControl.Advantech.Persistence
```

由上層 Application 組合。

## 為什麼不能放在核心層？

核心層的責任是：

- 控制
- 狀態
- I/O
- Native 隔離
- 安全策略

持久化屬於：

- 環境設定
- 現場校正值
- 軸參數
- Motion profile
- 軟體層組態

這是「系統組態問題」，不是「控制問題」。

一旦核心層開始：

- 讀 JSON
- 讀 XML
- 存資料庫
- 存檔

你會破壞：

- 可預測性
- 可測試性
- 核心穩定性

Level 5 架構必須讓核心層：

> 不知道資料從哪來
> 也不知道資料存去哪

## 正確架構分層

### 建議最終分層

Application
↓
Calin.MotionControl.Advantech.Configuration（新增）
↓
Calin.MotionControl.Advantech.PCI / EtherCAT
↓
Calin.MotionControl.Advantech

## 參數種類必須先分類

這是最重要的部分。

### 1️⃣ 硬體能力（Capability）

- 不可持久化
- 由 Factory 注入
- Immutable
- 來自型號定義

例如：

MaxVelocity
SupportsRdySignal

這些不是現場調整值。

### 2️⃣ 運動參數（Runtime Config）

例如：

- Acc
- Dec
- HomeSpeed
- JogSpeed
- SoftLimit
- GearRatio
- PulsePerUnit

這些應該：

- 可持久化
- 可修改
- 重新啟動後恢復

👉 這層才需要持久化

### 3️⃣ 運行時狀態（State）

例如：

- CurrentPosition
- IsHomed
- AlarmState

❌ 不應持久化
這是 runtime 狀態

## Level 5 正確做法

新增：

```csharp
IAxisConfiguration
ICardConfiguration
IDeviceConfiguration
```

核心層只接受：

```csharp
ApplyConfiguration(IAxisConfiguration config)
```

但：

- 不知道 config 來自哪裡
- 不負責保存

## Persistence 層該做什麼？

在 `Calin.MotionControl.Advantech.Configuration` 中：

- IParameterStore
- JsonFileParameterStore
- DatabaseParameterStore
- InMemoryParameterStore（測試用）

由 Application：

1. 讀取持久化
2. 建立 Configuration object
3. 注入 Device/Axis
4. 呼叫 ApplyConfiguration()

## 為什麼這是 Level 5 做法？

因為：

✔ 控制核心不依賴 IO
✔ 可單元測試
✔ 可替換儲存媒體
✔ 可支援未來 PLC/SCADA
✔ 不污染 I/O 執行緒
✔ 不影響即時性

## 若你硬要放在 PCI 子專案會發生什麼？

常見後果：

- Polling 中偷偷寫檔
- Stop 中保存參數
- GC 壓力增加
- 硬碟卡頓影響 I/O

工控系統中：

> 控制層絕對不能被磁碟 I/O 影響

## 進階建議（真正 Level 5）

加入：

### Configuration Snapshot

啟動時：

- 載入完整設定
- 生成 Immutable Snapshot
- 運行中不可修改

若需修改：

- 進入 Maintenance 模式
- 停止軸
- 重新套用設定

這是工控可預測性的來源。

## 最終建議架構

```csharp
Calin.MotionControl.Advantech              ← 核心控制（無持久化）
Calin.MotionControl.Advantech.PCI
Calin.MotionControl.Advantech.EtherCAT
Calin.MotionControl.Advantech.Configuration    ← 參數持久化
Application                          ← 組合層
```

## 如果你想做到更極致

可以再加入：

```csharp
Calin.MotionControl.Advantech.Profile
```

專門管理：

- Motion Profile
- Recipe
- 軸群組參數

讓控制核心保持最乾淨。

## 簡單結論

參數持久化：

✔ 不屬於核心
✔ 不屬於子類
✔ 屬於獨立組態層
✔ 由 Application 組裝
✔ 核心只接受 Immutable Configuration

# 參數持久化 (Copilot)

是，建議獨立專案。
以你現在要做的架構來看，參數持久化不應放在核心父類庫本體。

## 建議結論

### 應放在 Core 的

只放抽象與模型：
- IParameterStore
- IAxisConfiguration
- ICardConfiguration
- IDeviceConfiguration
- IControllerConfiguration
- ApplyConfiguration(...)
- ApplyParameters(...)
- 驗證器、不可變設定模型、版本欄位

### 不應放在 Core 的

所有具體儲存實作都不該放進來：
- JSON
- XML
- INI
- SQLite / DB
- NVRAM
- Registry
- 檔案路徑處理
- 序列化框架相依套件

## 原因

### 1. 保持 Core 純淨

你的 Calin.MotionControl.Advantech.Core 是父類核心庫，應該是：
- 硬體無關
- 儲存媒介無關
- App 無關

若把持久化實作放進 Core，Core 就會開始依賴：
- Newtonsoft.Json
- System.Text.Json
- DB provider
- 檔案格式
- 路徑與部署規則

這會讓 Core 變胖，也破壞長期穩定性。

### 2. 持久化需求本來就會變

未來常見情況：
- A 專案用 JSON
- B 專案用 SQLite
- C 專案用設備 NVRAM
- D 專案用 PLC/遠端設定中心

這些都不應回頭修改 Core。

### 3. 更適合 DI / Module 化

持久化天然適合獨立模組：
- JsonParameterStore
- SqlParameterStore
- NvramParameterStore
- MemoryParameterStore

App 在 Composition Root 決定注入哪個實作即可。

## 最佳分法

我建議至少拆成這樣：

```text
Calin.MotionControl.Advantech.Core
    ├─ 抽象介面
    ├─ Configuration models
    ├─ IParameterStore
    └─ validators

Calin.MotionControl.Advantech.Persistence.Json
    ├─ JsonParameterStore
    ├─ JsonConfigurationRepository
    └─ JSON 序列化實作

Calin.MotionControl.Advantech.Persistence.Sqlite
    ├─ SqliteParameterStore
    └─ DB 實作

Calin.MotionControl.Advantech.Persistence.Memory
    └─ MemoryParameterStore
```

## 需不需要再拆一層 Abstractions？

### 選項 A：介面放在 Core

最實用，也最簡單：
•	IParameterStore
•	IAxisConfiguration
•	ICardConfiguration

都放在 Calin.MotionControl.Advantech.Core
這通常就夠了。

### 選項 B：再拆 Abstractions

只有在你想讓很多專案只共用設定契約、但不想引用整個 Core 時才需要：
•	Calin.MotionControl.Advantech.Abstractions
•	Calin.MotionControl.Advantech.Core
•	Calin.MotionControl.Advantech.Persistence.Json

以你現在階段，通常先不用拆這麼細。

## 建議做法

•	Core：保留 IParameterStore 與 configuration contracts
•	Persistence：獨立專案做實作
•	App：決定用哪個 persistence provider
•	PCI / EtherCAT / Standalone：不要直接依賴具體 persistence 實作

## 一句話版本

是，應該獨立專案；Core 只保留持久化抽象，不保留持久化實作。
