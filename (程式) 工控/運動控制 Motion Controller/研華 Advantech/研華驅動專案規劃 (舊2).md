# 研華（Advantech）運動控制產品分類與 ACM/共通 API

## 主要運動控制產品類別

1. **PCI/PCIe 運動控制卡（集中式控制）**
    4/8 軸脈衝型卡，用於多軸伺服/步進馬達控制
    驅動直接透過 PCI/PCIe 插槽連接至 PC 進行即時運動控制。

    PC 或嵌入式控制平台結合 SoftMotion 控制核心

# 研華（Advantech）運動控制產品 **主要分類與介面／通訊**

## 1. **PC‑Based 可程式運動控制器**

- **AMAX‑357**
    4 核心 Atom 平台、32 軸 EtherCAT 運動控制器（SoftMotion + EtherCAT 主站）
    使用 MotionNavi API 開發環境與 Common Motion/SDK 支援。

- **MVP‑3245**
    嵌入式平台，具線性、圓弧插補、E‑Gear、E‑CAM 等進階運動功能，同樣支援 Common Motion API。

## 2. **EtherCAT 運動控制／通訊卡（PC 主站／主設備）**

這些產品多為 **PCI / PCIe EtherCAT master card**，用於控制 EtherCAT 從站（伺服驅動、I/O 模組等）：

- **PCI‑1203 系列（PCI）**
    EtherCAT PCI 主卡，提供 2 個 EtherCAT 埠
    支援多達 32 軸運動控制（依型號可到 6/10/16/32 軸）。

- **PCIE‑1203 / PCIE‑1203IO / PCIE‑1203L（PCI Express）**
    EtherCAT PCIe 主卡
    PCIE‑1203：EtherCAT 主站 + 運動控制
    PCIE‑1203IO：EtherCAT 主站但側重 I/O
    PCIE‑1203L：支援更多通道（如 64 軸等），依不同配置。

- **PCM‑26R1EC**
    ARM 基底 Mini PCIe EtherCAT 裝置（MDevice 模式），可作為 EtherCAT 裝置執行支援。

## 3. **Standalone / ARM 基底控制器**

- **AMAX‑3110 等 ARM 版控制器**
    具本地 SoftMotion 或 Pulse Train 控制等功能
    可離線執行控制邏輯（不需 PC 主機）。

## 4. **Motion Control Modules（介面模組／從站）**

研華提供 EtherCAT 從站模組，可連接伺服/步進驅動器或編碼器等：

- **AMAX‑3245 / AMAX‑3245C**： 4 軸脈衝 EtherCAT 從站模組
- **AMAX‑4270**： 編碼器/計數器模組
    這些模組常與 EtherCAT 主站卡（如 PCI‑1203 族）搭配使用。

## 5. **傳統 PCI 運動控制卡（非 EtherCAT）**

早期研華也有傳統脈衝／DSP 型 PCI 運動卡，例如：

- **PCI‑1220U / PCI‑1240U**：2 軸／4 軸脈衝型運動控制卡
    這類較少用於 EtherCAT，而是直接透過脈衝輸出與伺服驅動器/步進驅動器連線。

# 以「工控軟體開發」角度分類研華運動控制產品線

如果從 **寫驅動 / 使用 SDK / LIB 的角度** 來看，研華產品其實不是依硬體型號分類，而是依 **Driver 架構 + API Library** 分類。
這樣的分類對於設計 **統一 Motion Control Library**（例如你的 `Calin.MotionControl.Advantech`）最合理。

最實務的分類只有 **4 類**。

## 1 脈衝型 Motion Card（ACM / Motion API）

**特徵**

- PC 插卡
- 直接輸出 Pulse / Dir
- 軸控制在卡上 DSP

**硬體**

- PCI-1240U
- PCI-1220U
- PCI-1265
- PCI-1285

**Driver / LIB**

- Advantech Motion Driver
- **ACM (Advantech Common Motion API)**

**程式特性**

- Axis handle
- Card handle
- DSP-based motion
- 呼叫類似：

```csharp
Motion_DevOpen
Motion_AxMoveAbs
Motion_AxMoveRel
Motion_AxStop
```

**軟體模型**

```csharp
Application
    ↓
ACM Library
    ↓
PCI Motion Card
    ↓
Servo Driver
```

## 2 EtherCAT Motion（MotionNavi / SoftMotion）

**特徵**

- PC 當 EtherCAT Master
- 多軸同步
- 分散式伺服

**硬體**

- PCIE-1203
- PCI-1203
- AMAX EtherCAT

**Driver / LIB**

- **MotionNavi SDK**
- EtherCAT stack

**程式特性**

- Node
- PDO / SDO
- Distributed clock

**軟體模型**

```csharp
Application
    ↓
MotionNavi API
    ↓
EtherCAT Master
    ↓
Servo Drive
```

## 3 PC-Based Motion Controller（SoftMotion Controller）

**特徵**

- 控制器本身執行 Motion Kernel
- PC 只是 HMI / 上位機

**硬體**

- AMAX-357
- MVP 系列

**Driver / LIB**

- MotionNavi
- Controller SDK

**程式特性**

```csharp
Application
    ↓
Controller API
    ↓
Motion Kernel
    ↓
EtherCAT Servo
```

## 4 EtherCAT Slave Motion Module

**特徵**

- 不直接由 PC 控制
- 必須透過 EtherCAT master

**硬體**

- AMAX-3245
- AMAX-4270

**Driver**

- EtherCAT protocol

**軟體模型**

```csharp
Application
    ↓
EtherCAT Master
    ↓
AMAX Module
```

## 工控程式設計最合理的分類

如果你是做 **工控驅動封裝庫**，建議使用這個分類：

```csharp
AdvantechMotion
│
├─ PulseMotion
│   ├─ ACM Driver
│   └─ PCI Motion Cards
│
├─ EtherCATMotion
│   ├─ MotionNavi
│   └─ EtherCAT Master Cards
│
├─ PCBasedController
│   └─ AMAX / MVP
│
└─ EtherCATModules
    └─ AMAX Slave Modules
```

## 最佳架構（推薦）

建議不要依 **PCI / EtherCAT / 型號** 分。

而是依 **Driver / Library 架構**：

```csharp
Calin.MotionControl.Advantech
│
├─ Calin.MotionControl.Advantech.Core (核心層)
│
├─ Calin.MotionControl.Advantech.Configuration (參數持久化層)
│
├─ Calin.MotionControl.Advantech.ACM
│   └─ Pulse Motion Cards
│
├─ Calin.MotionControl.Advantech.MotionNavi
│   └─ EtherCAT Master
│
├─ Calin.MotionControl.Advantech.Controller
│   └─ AMAX / MVP
│
└─ Calin.MotionControl.Advantech.EtherCAT
    └─ Slave Modules
```

這樣的好處：

- 完全對齊 **研華 SDK**
- 新產品不會破壞架構
- LIB 層可以共用

## 一個非常重要的事（很多人會踩坑）

研華其實有 **兩個完全不同的 Motion SDK 生態系**：

|SDK|產品|
|---|---|
|**ACM (Common Motion)**|傳統 PCI Motion|
|**MotionNavi**|EtherCAT / Controller|

兩者 **API 完全不同**。

所以你的 Core 設計一定要 **抽象 Axis / Controller**，不要直接綁 API。

## 如果是「專業工控架構」

真正成熟的 Motion Library 會長這樣：

```csharp
Application
     ↓
Motion Abstraction Layer
     ↓
Motion Provider
     ├─ Advantech ACM
     ├─ Advantech MotionNavi
     ├─ Beckhoff TwinCAT
     ├─ Delta Motion
     ↓
Hardware
```

這樣你的系統就會變成 **Motion Framework**。

# 工控級 Level 5 架構（完整規劃）

目標是建立 **可長期維護、可擴充、可模擬、可跨設備的運動控制框架**，適用於：

- 24/7 工控設備
- Windows 7 低階電腦
- 長期設備維護（10~15 年）
- 多種研華運動控制產品

設計核心原則：

- **Core 抽象化**
- **Driver 與 API Library 分離**
- **Configuration 參數持久化**
- **Runtime 管理生命週期**
- **Simulation 可離線開發**
- **APP 完全不依賴 Driver**

整體架構：

```csharp
Calin.MotionControl.Advantech
│
├─ Calin.MotionControl.Advantech.Core
│
├─ Calin.MotionControl.Advantech.Configuration
│
├─ Calin.MotionControl.Advantech.Runtime
│
├─ Calin.MotionControl.Advantech.Simulation
│
├─ Calin.MotionControl.Advantech.ACM
│
├─ Calin.MotionControl.Advantech.MotionNavi
│
├─ Calin.MotionControl.Advantech.Controller
│
└─ Calin.MotionControl.Advantech.EtherCAT
```

APP：

```csharp
Application
    ↓
Core + Runtime
    ↓
Driver / Simulation
```

APP **永遠只依賴 Core / Runtime / Configuration**。

Calin.MotionControl.Advantech.Simulation

### 目標

允許 **沒有硬體也能運行系統**。

用於：

- UI 開發
- 流程測試
- 自動測試
- Demo

### 職責

1. 模擬 Axis
2. 模擬 Motion
3. 模擬 IO
4. 模擬 Alarm
5. 模擬 Servo

### 主要類別

Device

```csharp
SimulatedMotionDevice
```

Axis

```csharp
SimulatedAxis
```

IO

```csharp
SimulatedIo
```

### Motion 模型

位置計算：

```csharp
Position += Velocity * dt
```

狀態模擬：

```csharp
Moving
Stopped
Homing
Alarm
```

### Simulation 優點

- 可離線開發
- UI 與硬體解耦
- 可做 CI 測試

## 工控級 Level 5 設計要求

24/7 系統需滿足：

### 記憶體穩定

避免：

```csharp
LINQ
大量 allocation
closure
boxing
```

### Driver 單執行緒

所有 Motion API 建議：

```csharp
Single Motion Thread
```

### Polling 架構

集中輪詢：

```csharp
MotionPollingService
```

週期：

```csharp
10 ms
```

### Fault Recovery

Driver 必須支援：

```csharp
Reconnect
Reset
Alarm clear
Reinitialize
```

### Logging

建議加入：

```csharp
MotionLog
DriverLog
FaultLog
```
