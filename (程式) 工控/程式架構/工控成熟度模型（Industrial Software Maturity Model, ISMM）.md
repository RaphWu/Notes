# 工控成熟度模型（Industrial Software Maturity Model, ISMM）

本模型適用於**所有工控軟體**，不限通訊、設備控制、資料處理、UI 或整體系統。
目標：建立可量化、可審核、可長期演進的 24/7 工控軟體成熟度標準。

核心原則：

1. 穩定性優先於功能
2. 可預測性優先於彈性
3. 故障可隔離優先於自動隱藏錯誤
4. 明確生命週期管理
5. 長時間運作無資源成長

成熟度分為 Level 1～5。

## Level 1 — 功能可運作（Functional）

### 定義

- 功能完整
- 在實驗環境可正常執行

### 必要條件

- 主要功能可操作
- 基本例外處理
- 可正常啟動與關閉

### 尚未要求

- Thread-safe
- 記憶體穩定性
- 並發控制
- 錯誤分類

### 適用

- Demo
- 原型
- 實驗驗證

## Level 2 — 穩定應用（Stable Application）

### 定義

- 可長時間（8~24 小時）運作
- 無明顯資源洩漏

### 必須具備

- 所有背景工作可安全取消
- Dispose 可重入
- 無未捕捉例外
- Thread-safe public API
- 不使用 Thread.Abort
- 所有 I/O 有 Timeout

### 仍可能缺乏

- 嚴格並發隔離
- GC 控制
- 狀態可觀察性
- 健康回報機制

## Level 3 — 強健工控系統（Robust Industrial）

### 定義

- 可連續運作數天以上
- 並發安全
- 錯誤模型清晰

### 強制要求

- 所有共享資源有明確同步策略
- 無 race condition
- 無 deadlock
- 所有錯誤分類清晰（不得簡化為單一錯誤）
- 狀態轉換明確（State Machine）
- 不允許 out-of-order 行為
- 記憶體無成長趨勢
- 日誌分級（Trace/Debug/Warning/Error）

### 適用

- 產線設備
- 長時間連續班運轉

## Level 4 — 24/7 工控級（Continuous Industrial Grade）

### 定義

- 可長時間（數月）不重啟
- 記憶體與 GC 可預測
- 故障可隔離

### 強制要求

- 所有模組具備明確生命週期（Initialize / Start / Stop / Dispose）
- 不允許隱性背景執行緒
- 所有 Task 必須綁定 CancellationToken
- 事件不可阻塞 I/O 或控制主線程
- 可提供 HealthReport
- 可觀察：
  - 最後成功時間
  - 連續失敗次數
  - Offline/Degraded 狀態
- 支援壓力測試（100+ 並發情境）
- 高頻路徑避免多餘配置與 GC 壓力

### 驗證條件

- 長時間 soak test
- 記憶體曲線平穩
- CPU 使用率可預測
- 無資源洩漏（Handle / GDI / Socket）

## Level 5 — 關鍵基礎設施級（Mission-Critical Industrial）

### 定義

- 可作為產線核心或基礎設施
- 故障不會導致系統失控
- 行為完全可預測

### 強制要求

- 嚴格狀態機
- 所有錯誤具明確恢復策略
- 明確 Offline → Recovery → Online 流程
- 不允許自動隱藏重大錯誤
- GC 行為經壓測驗證
- 所有模組可單獨隔離測試
- 可支援多實例並行
- 所有資源使用可量測
- 發版前必須通過：
  - 並發壓力測試
  - 長時間穩定測試
  - 邊界條件測試
  - 故障注入測試

| ---------------- | --- | --- | --- | --- | --- |

| Thread-safe API  | ✘   | ✔   | ✔   | ✔   | ✔   |
| 背景 Task 可取消 | ✘   | ✔   | ✔   | ✔   | ✔   |
| 並發安全         | ✘   | ✘   | ✔   | ✔   | ✔   |

| 記憶體穩定       | ✘   | ✘   | ✔   | ✔   | ✔   |

| 故障注入測試     | ✘   | ✘   | ✘   | ✘   | ✔   |
| 行為完全可預測   | ✘   | ✘   | ✘   | ✘   | ✔   |

## 工控軟體通用強制規則（所有 Level ≥3 必須遵守）

1. 不得有未處理例外
2. 不得有未釋放資源
3. 不得依賴重啟修復問題
4. 不得使用非受控終止機制
5. 不得允許無限重試
6. 不得使用隱性全域狀態
7. 所有 I/O 必須可 Timeout
8. 所有背景行為必須可取消

## 建議實務使用方式

1. 每個專案明確標註目標成熟度
2. 未達 L3 不可進產線
3. 未達 L4 不可 24/7 運作
4. 核心控制系統必須達 L5

## 簡化判斷原則

如果系統：

- 可以重啟 → 通常 ≤ L3
- 不允許重啟 → 必須 ≥ L4
- 發生錯誤會自動隱藏 → ≤ L2
- 錯誤必須可觀察與分類 → ≥ L3
- 記憶體曲線必須平穩 → ≥ L4

# 工控級成熟度發版審核清單（ISMM Release Gate Checklist）

本清單用於所有工控軟體專案發版前審核。
適用範圍：設備控制、通訊模組、資料處理、WinForms/HMI、背景服務、整體控制系統。

使用方式：

- 每次發版前逐條審核
- 未達目標成熟度不得釋出
- 建議納入 CI 或發版簽核流程

## 一、基本發版門檻（所有等級必須通過）

1. 無未處理例外
2. 所有背景 Task 可安全取消
3. Dispose 可重入且釋放全部資源
4. 所有 I/O 具 Timeout
5. 無 Thread.Abort
6. 無無限重試
7. 無 Service Locator
8. 無隱性背景執行緒
9. 日誌等級分級明確
10. 例外不可直接吞掉

## 二、Level 3（Robust Industrial）發版條件

### 並發安全

1. 所有共享資源有明確同步機制
2. 無 race condition
3. 無 deadlock
4. 不允許 out-of-order 行為

### 錯誤模型

5. 所有錯誤具明確分類
6. 不得簡化為單一錯誤類型
7. 錯誤不可依賴重啟解決
8. 重試條件可控且可停止

### 記憶體

9. 無長時間記憶體成長趨勢
10. 無未釋放 Handle / GDI / Socket

## 三、Level 4（24/7 工控級）發版條件

### 生命週期管理

1. 所有模組具明確 Initialize / Start / Stop / Dispose
2. 未 Initialize 不可執行核心功能
3. Dispose 後所有 API 明確拒絕

### 健康監控

4. 可提供 HealthReport
5. 可查詢最後成功時間
6. 可查詢連續失敗次數
7. 可查詢 Offline / Degraded 狀態

### 長時間驗證

8. 通過 72 小時以上 soak test
9. 記憶體曲線平穩
10. CPU 使用率可預測
11. 無例外爆量日誌

### 高頻效能

12. 高頻路徑無多餘配置
13. 無不必要 GC 壓力
14. 無阻塞 I/O 主線程

## 四、Level 5（Mission-Critical）發版條件

### 狀態機

1. 所有核心模組具嚴格狀態機
2. 狀態轉換可追蹤
3. Offline → Recovery → Online 流程明確

### 故障隔離

4. 單一模組故障不影響整體系統
5. 故障不會導致失控行為
6. 錯誤不會自動隱藏

### 壓力與故障測試

7. 通過並發壓力測試
8. 通過邊界條件測試
9. 通過故障注入測試
10. 通過長時間 100+ 實例並行測試

### 可量測性

11. 所有關鍵資源使用可監測
12. GC 行為經壓測驗證
13. 日誌不會在高頻路徑造成效能問題

## 五、WinForms / HMI 專用審核項目（若適用）

1. UI 執行緒不得被背景工作阻塞
2. 所有跨執行緒操作使用安全封裝
3. 無 GDI 洩漏
4. 視窗開關不造成 Handle 成長
5. 長時間執行無 UI Freeze

## 六、資料與檔案處理專用審核（若適用）

1. 檔案操作具例外處理
2. 不會產生無限檔案成長
3. 資料寫入具完整性保護
4. 不因例外造成資料損毀

## 七、通訊模組專用審核（若適用）

1. 封包驗證完整
2. 錯誤分類精細
3. 不允許隨機重試
4. 並發序列化控制正確
5. 長時間無封包殘留記憶體

## 八、量化發版門檻建議

| 等級 | 最低 Soak Test | 並發測試 | 故障注入 |
| ---- | -------------- | -------- | -------- |
| L3   | 24 小時        | 基本     | 不要求   |
| L4   | 72 小時        | 50+      | 基本     |
| L5   | 168 小時       | 100+     | 必須     |

# 工控級 vs 企業級

這兩個詞**有重疊，但不是同一個維度**。
差異在「設計優先順序」與「風險容忍度」。

下面用工程觀點精確區分。

## 一、企業級（Enterprise-grade）

### 核心目標

- 可維護性
- 可擴充性
- 多團隊協作
- 長期版本演進
- 跨系統整合

### 常見特徵

- DI / IOC 容器
- Repository / Service 分層
- 日誌與監控整合
- 可插拔架構
- 雲端部署友善
- API 版本控制
- CI/CD 導向

### 設計優先順序

1. 可維護性
2. 擴充性
3. 清晰分層
4. 測試覆蓋率
5. 效能（合理即可）

### 容忍

- 小量 GC
- 偶發重啟
- Rolling restart
- 短暫不可用

## 二、工控級（Industrial / 24x7 Control-grade）

### 核心目標

- 絕對穩定
- 可預測
- 長時間不重啟
- 不可失控

### 常見特徵

- 嚴格 thread-safe
- 精準錯誤分類
- 無記憶體洩漏
- 可取消背景 Task
- 無 out-of-order
- 明確狀態機
- 不允許未定義行為

### 設計優先順序

1. 穩定性
2. 可預測性
3. 效能（低延遲 + 低 GC）
4. 錯誤隔離
5. 擴充性

### 不容忍

- Deadlock
- Race condition
- 未處理例外
- 自動重啟掩蓋錯誤
- 記憶體成長
- Thread.Abort

## 三、關鍵差異對照

| 面向      | 企業級    | 工控級       |
| --------- | --------- | ------------ |
| 目標      | 可演進    | 不當機       |
| GC 壓力   | 可接受    | 必須控制     |
| Restart   | 可接受    | 通常不允許   |
| Exception | 可 bubble | 必須消化     |
| 設計風格  | 彈性      | 嚴謹         |
| 抽象層    | 偏多      | 僅必要       |
| AutoRetry | 常見      | 需精準分類   |
| 日誌      | 可大量    | 高頻路徑限制 |

## 四、在你這個 Modbus 核心庫的情境

你其實要的是：

> 工控級穩定性 + 企業級架構

也就是：

- 協定層 → 工控級
- 架構層（DI / 三層 / 可擴充）→ 企業級
- 記憶體與並發 → 工控級
- 版本管理與專案分離 → 企業級

## 五、簡單判斷公式

如果系統：

- 可以重啟 → 企業級
- 不能重啟 → 工控級

如果錯誤：

- 可以重試掩蓋 → 企業級
- 必須精準分類 → 工控級

如果 GC：

- 不是關鍵 → 企業級
- 需要可預測 → 工控級

## 六、你的系統屬於哪種？

根據你之前討論的：

- 24/7
- 100+ 設備
- 不可記憶體洩漏
- 明確 Exception 分類
- 不允許 out-of-order

這是**工控級核心 + 企業級封裝**。

不是純企業系統。
